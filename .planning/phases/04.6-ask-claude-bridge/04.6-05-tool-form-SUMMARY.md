## Accomplishments

- **D-05 structured args:** input schema declares `{question (required, 1-4000 chars), tried (required, 10-4000 chars), relevant_context (optional, ≤ 8000 chars)}` with `additionalProperties: false`. The `tried` field's `minLength: 10` is a structural floor; the `_BOILERPLATE_TRIED_PATTERNS` regex set is the substantive gate that filters short-but-junk values like "stuck" / "n/a" / "I tried thinking about it" / "I don't know …" / pure whitespace before any sidecar dispatch.
- **D-13 opt-in per profile:** `createAskClaudeTool` returns `null` when `config.enabled !== true`. Default state on every shipping bundle (no `tools.ask_claude` block, or block with `enabled: false`) leaves the tool unregistered — no harness behavioral surface for the model. Test `returns null when config.enabled is false` asserts the factory contract.
- **D-15 X circumstances verbatim:** `TOOL_DESCRIPTION` is a 27-line string copy-pasted verbatim from PLAN lines 190-218 — the same operator-authored "Use ask_claude WHEN / Do NOT use ask_claude / Format your call / Per-turn limit: 5 calls" block. Tests assert the description contains `Use ask_claude WHEN`, `Do NOT use ask_claude`, `secrets, credentials, PII`, and `per-turn limit` — guards against any future paraphrase drift.
- **D-07 per-turn rate-limit:** module-level `_turnAskClaudeCount` checked BEFORE increment (gate-before-bump pattern from web-search.ts). 6th call in a turn returns a structured `ask_claude rate-limit (5 per-turn) exceeded` ToolError; `callAskClaude` is NOT invoked for the over-budget call (test `6th call in a turn blocked with structured rate-limit error` asserts both the surface and the underlying-call non-invocation). `resetAskClaudeTurnCount()` is exported for `pi.on('turn_end', ...)` wiring.
- **T-06 boilerplate-tried discipline gate:** `_BOILERPLATE_TRIED_PATTERNS` is two regexes — an exact-equality set (anchored both ends) for short tokens ("stuck", "n/a", "none", "nothing", "tried", "thinking", whitespace) and a prefix set (anchored at `^` only) for long-form openers ("I tried thinking ...", "I don't know ..."). The split is the Rule-1 bug fix surfaced by the RED suite — the PLAN's single combined regex with `\\s*$` would have excluded "I tried thinking about it". Validation rejects do NOT consume rate-limit headroom (regression-tested by `counter only increments on successful dispatch, not on validation rejects`).
- **Sidecar error verbatim surface:** `callAskClaudeViaSidecar` maps 4xx/5xx responses to `AskClaudeError` with `reason` / `pattern_class` / `matched_excerpt` / `detail` copied from the JSON body. The tool's `invoke` handler then surfaces those fields directly: `ask_claude failed: scrubber_blocked (sk_prefixed_key). [optional detail]`. The model sees *which* class of pattern was blocked so it can self-correct without re-attempting the same forbidden category.
- **CLAUDE.md URL precedence:** `callAskClaudeViaSidecar` resolves baseUrl as `opts.baseUrl ?? process.env.EMMY_SERVE_URL ?? "http://127.0.0.1:8003"` with trailing-slash strip — identical pattern to `profile-swap-runner-http.ts` and `metrics-poller.ts`. Profile-side overrides flow through `opts.baseUrl` from the future session.ts wiring (deferred — outside this plan's scope).

## Decisions Made

- **Mirror web-search.ts surface (PiToolDefinitionShape) over plan's pseudocode shape (Tool with execute):** the plan's pseudocode uses `{input_schema, execute}` (Anthropic Messages-API tool shape), but the existing harness wiring (session.ts → toolSpecToDefinition → pi.customTools) expects `{name, description, parameters, invoke}`. Adopting PiToolDefinitionShape keeps the wire-through path uniform with web-search and the future Plan 04.6-04 slash-command consumer doesn't need a parallel adapter. Test assertions adapted to the actual shape (`tool.parameters.properties.*` instead of `tool.input_schema.properties.*`, `tool.invoke(...)` instead of `tool.execute(...)`). Functional contract unchanged — both shapes carry the same {schema, dispatcher} pair.
- **Boilerplate-tried regex split into exact-equality + prefix sets:** PLAN's combined regex `^\\s*(...|i tried thinking|...)\\.?\\s*$/i` would have excluded "I tried thinking about it" (test corpus item) because the trailing `\\s*$` anchor requires the regex to match the entire trimmed string. Split fix: short tokens still anchor both ends ("stuck", "n/a", etc.), but long-form openers anchor at `^` only ("i tried thinking" / "i don't know" prefixes catch any continuation). Surfaced by the RED suite (test `boilerplate tried rejected` failed with "should not be called" — meaning the dispatch went through to the stub).
- **Counter increments AFTER successful sidecar dispatch, not on every invoke:** validation rejects (boilerplate-tried) and rate-limit gates short-circuit before the counter bump. Two regression invariants: (a) over-budget call doesn't burn more headroom (PLAN's pattern, mirrors web-search); (b) 10 boilerplate-tried calls don't burn 5-per-turn budget (regression test added). The combined behavior keeps the per-turn budget meaningful for the model — it reflects ACTUAL Claude consultations, not validation churn.
- **callAskClaudeViaSidecar shipped in this plan even though its consumer is downstream:** keeps the type contract single-sourced (the AskClaudeError shape is defined alongside the producer that throws it). URL config precedence (env > profile > 127.0.0.1:8003) implemented per CLAUDE.md verbatim; trailing-slash strip mirrors profile-swap-runner-http.ts. The function is exported from @emmy/tools but the only required wire-up step in session.ts will be `callAskClaude: (prompt) => callAskClaudeViaSidecar(prompt, { baseUrl: profileBaseUrl })`.
- **Error message text reshaped to satisfy operator-authored test regexes:** the test suite's regexes (`/tried.*concrete prior attempts/i`, `/rate.?limit.*per.?turn/`) are the surface contract — reshaping the wording so they match is non-behavioral. New texts: `` "`tried` field requires concrete prior attempts. ..." `` and `"ask_claude rate-limit (5 per-turn) exceeded. ..."`. Both keep the operator-suggested actionable hint ("Try reading more files, running tests..." / "Continue the work locally and try again on a future turn.").

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Boilerplate-tried regex excluded long-form prefixes ("I tried thinking about it")**

- **Found during:** Task 2 GREEN verification (RED suite test `boilerplate tried rejected` failed with `should not be called` — meaning the dispatch went through to the stub; the regex didn't match)
- **Issue:** PLAN's combined regex `/^\\s*(n\\/?a|none|nothing|tried|stuck|i don'?t know|i tried thinking|thinking)\\.?\\s*$/i` requires the trailing `\\s*$` anchor to bind. "I tried thinking about it" has " about it" after "thinking" — the `\\s*$` fails. So the regex only catches the exact form "I tried thinking" (with optional trailing dot/whitespace), not the more typical conversational form.
- **Fix:** Split into two regexes — exact-equality set (anchored both ends) and prefix set (anchored at `^` only). Long-form openers ("i tried thinking" / "i don't know") now match any continuation.
- **Files modified:** `packages/emmy-tools/src/ask-claude.ts` (`_BOILERPLATE_TRIED_PATTERNS` constant)
- **Verification:** All 9 boilerplate corpus items now caught; 16/16 tests pass.
- **Committed in:** `ea9146a` (Task 2 GREEN — fix is part of the implementation commit since the regex was first introduced there)

**2. [Rule 1 - Bug] Error message text didn't satisfy test regex contracts**

- **Found during:** Task 2 GREEN verification (RED suite tests `empty tried field rejected` and `6th call in a turn blocked` failed because text-shape didn't match expected regexes)
- **Issue:** PLAN's surface text put `concrete prior attempts` BEFORE `tried` in the boilerplate-error string, but the test regex `/tried.*concrete prior attempts/i` requires the reverse order. Similarly, the rate-limit error said `per-turn rate-limit (N/turn) exceeded` but the test regex `/rate.?limit.*per.?turn/` (run after `.toLowerCase()`) requires `rate-limit` before `per-turn`.
- **Fix:** Reshaped both error strings:
  - Boilerplate: `` "`tried` field requires concrete prior attempts. ..." ``
  - Rate-limit: `"ask_claude rate-limit (5 per-turn) exceeded. ..."`
- **Files modified:** `packages/emmy-tools/src/ask-claude.ts` (two `_err(...)` calls in `invoke`)
- **Verification:** Both regexes now match; 16/16 tests pass.
- **Committed in:** `ea9146a` (Task 2 GREEN — same commit as the regex fix; both are surface-text adjustments)

**3. [Rule 3 - Blocking] Pydantic-style `Tool` shape replaced with PiToolDefinitionShape**

- **Found during:** Task 2 GREEN scaffolding
- **Issue:** PLAN's pseudocode uses an Anthropic Messages-API-style tool shape (`{name, description, input_schema, execute}` returning `{is_error, content}`). The existing harness wiring (web-search.ts → session.ts → toolSpecToDefinition / registerNativeTools → pi 0.68's `customTools`) expects PiToolDefinitionShape (`{name, description, parameters, invoke}` returning `unknown`). Adopting the Anthropic shape would have required a parallel adapter and broken byte-stability with web-search.
- **Fix:** Implemented the factory as PiToolDefinitionShape — same surface as web-search. Test assertions retargeted: `tool.input_schema.properties.*` → `tool.parameters.properties.*`, `tool.execute(...)` → `tool.invoke(...)`, `result.is_error` → `result.isError` (the convention the existing tools use), `result.content` → `result.content[0].text` (pi's structured content array). Functional contract unchanged.
- **Files modified:** `packages/emmy-tools/src/ask-claude.ts` (factory body) + `packages/emmy-tools/tests/ask-claude.test.ts` (assertion shapes — written upfront in RED to anticipate this).
- **Verification:** Tests authored against PiToolDefinitionShape pass; production wiring path is uniform with web-search (Plan 04.6-04 slash command + future session.ts wiring don't need parallel adapters).
- **Committed in:** `e4b3737` (RED — test assertion shape) + `ea9146a` (GREEN — factory body)
