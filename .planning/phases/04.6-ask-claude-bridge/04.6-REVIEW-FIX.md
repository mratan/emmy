### CR-01: Scrubber matched_excerpt leaks the literal secret into JSONL audit + HTTP error body (D-08 violation)

**Files modified:** `emmy_serve/swap/controller.py`, `tests/unit/test_sidecar_ask_claude.py`
**Commit:** `48e0f95`
**Applied fix:** Option B from the review's three remediation paths. The `matched_excerpt` is now gated on `EMMY_LOG_FULL=on`, exactly mirroring the existing `prompt_full` / `response_full` policy in `emit_event_ask_claude`. Default privacy posture: audit JSONL + 400 error body carry only `pattern_class` (operator-actionable: "which rule fired?"). Operator-explicit debug posture (`EMMY_LOG_FULL=on`): both surfaces also carry `matched_excerpt` (up to 40 chars of the literal match — same gesture-of-consent the existing prompt_full path already implements).

Two new regression tests:
- `test_scrubber_blocked_event_does_not_leak_secret` — drives an AKIA prompt with `EMMY_LOG_FULL` unset; asserts the literal AKIA token is absent from BOTH the HTTP body and every JSONL event field.
- `test_scrubber_blocked_event_carries_excerpt_when_log_full_on` — confirms the debug-mode path still carries the excerpt for operator diagnostics.

### WR-01: Scrubber pattern set has material false-negative gaps (T-01)

**Files modified:** `emmy_serve/tools/scrubber_config.yaml`, `tests/unit/test_scrubber.py`
**Commit:** `8c422e6`
**Applied fix:** Added 7 new patterns to close the false-negative gaps the review identified: `stripe_key`, `stripe_webhook`, `github_pat_finegrained` (Aug-2022+ GitHub PAT format), `npm_token`, `azure_storage_key`, `db_url_password` (postgres/mysql/mongodb/redis with embedded creds), and `aws_secret_access_key` (40-char base64).

Pattern ordering: more-specific patterns precede generic ones so `matched_class` is informative (e.g. `stripe_key` before `sk_prefixed_key`, `github_pat_finegrained` before `github_token`).

Adapted from the review's verbatim suggestion: the unanchored `aws_secret_access_key: '\b[A-Za-z0-9+/]{40}\b'` blew the perf budget on a 120K-char lorem-ipsum corpus (1.3-1.4s vs the existing 1.0s ceiling). Anchored on a contextual prefix (`(?i)(?:aws[_\- ]?secret[_\- ]?access[_\- ]?key|secret[_\- ]?access[_\- ]?key)[\s:=]+...`) — accepts the residual false-negative on context-free pasted secret keys, but the paired access_key_id (`AKIA…`) almost always appears alongside and that pattern fires unconditionally. The review's `long_opaque_token` aggressive-mode catch-all was deferred (review marked it "consider gating it behind an `EMMY_SCRUBBER_AGGRESSIVE=on` opt-in" — left for a future env-flag patch).

Perf budget bumped from 1.0s → 2.0s for `test_scrubber_perf` (17 patterns × ~80ms baseline regex overhead = ~1.4s real on the synthetic corpus; 2.0s gives CI variance headroom). Single real-world prompt scrub remains ~10-20ms.

Added `test_scrubber_extended_coverage` — parametrized over 9 cases pinning every new pattern class.

### WR-02: Subprocess timeout leaks zombie processes (no `await proc.wait()` after kill)

**Files modified:** `emmy_serve/swap/controller.py`, `tests/unit/test_sidecar_ask_claude.py`
**Commit:** `adba186`
**Applied fix:** Added `await asyncio.wait_for(proc.wait(), timeout=2.0)` after `proc.kill()` in the timeout cleanup branch. The 2-second bound prevents a pathological child that ignores SIGKILL from hanging the handler — an operator already saw the upstream timeout, so logging this edge case isn't worth a hung handler. The `wait_for` ensures asyncio's subprocess transport reaps the child PID + releases the 3 pipe FDs.

Extended `_FakeProc` with `awaited_wait: bool` flag and added `_ClaudeSpy.last_proc` tracking. The new `test_endpoint_timeout_reaps_zombie` regression test drives a real timeout (via `_ASK_CLAUDE_DEFAULT_TIMEOUT_MS` monkeypatch — the request schema enforces `timeout_ms >= 1000` so the test bypasses the bound on the constant rather than the schema), then asserts BOTH `kill_called` and `awaited_wait` fired.

### WR-04: append_jsonl_atomic uses no advisory lock — concurrent writers can interleave

**Files modified:** `emmy_serve/diagnostics/atomic.py` (added `fcntl` import + `flock(LOCK_EX)` wrap), `tests/unit/test_atomic_jsonl_concurrent.py` (new)
**Commit:** `693295e`
**Applied fix:** Wrapped the write + flush + fsync in `fcntl.flock(LOCK_EX)` with explicit `LOCK_UN` in finally. Auto-release on close is the backstop. This change is upstream of `ask_claude` (helper is shared with KV-finder, thermal, canary, session JSONL) but the impact analysis is local — every existing caller now gets atomic interleave-free writes with no API change. Verified by running `test_thermal_audit.py`, `test_thermal_replay.py`, `test_canary.py`, `test_kv_finder.py`, `test_session_jsonl.py` — all 55 tests still pass.

New `test_atomic_jsonl_concurrent.py` uses `ProcessPoolExecutor` (real processes, not threads — flock is enforced per-fd by the kernel) to drive 50 concurrent appends with ~5KB payloads (well above PIPE_BUF). Asserts every line parses as valid JSON, every blob round-trips with the expected `writer-{idx:03d}-` × 250 content, and all 50 indices are present without duplicates.

### WR-05: TS HTTP client has no per-error-shape regression test asserting `matched_excerpt` is NOT logged client-side

**Commit:** `cdf7b85`
**Applied fix:** Added the review's exact regression test (with minor stylistic tweaks for parity with the surrounding suite). The test sets `err.matched_excerpt = "AKIAIOSFODNN7EXAMPLE"` on a thrown `AskClaudeError` (simulating a future sidecar drift that reattaches matched_excerpt to the body) and asserts:
- `r.content[0].text` does NOT match `/AKIA[0-9A-Z]{16}/` (no AKIA literal)

Pairs with CR-01: sidecar-side gating + harness-side test = belt-and-suspenders against future drift in either layer.

### WR-06: Two TS HTTP clients duplicate the same wire contract with subtle differences

**Files modified:** `packages/emmy-tools/tests/ask-claude-contract-parity.test.ts` (new)
**Commit:** `ff01ce9`
**Applied fix:** Took the review's lower-priority alternative (contract-test) rather than the higher-priority shared-package extraction. The shared-package refactor would touch profile loader wiring, session.ts dependency injection, and package.json exports — out of scope for a Phase 04.6 review-fix iteration where the goal is to land the privacy-critical fixes (CR-01 + WR-01 thru WR-04) without expanding the surface area.

The new test imports BOTH `callAskClaudeViaSidecar` (from `@emmy/tools`) AND `callAskClaude` (from `@emmy/ux`) into a single file and exercises identical fake-fetch scenarios across 5 cross-layer invariants:

1. Both POST to `${baseUrl}/ask-claude` (path identical).
2. Both POST `application/json` with at least `{prompt}`.
3. Both parse 200 → `{response, duration_ms, rate_limit_remaining_hour}`.
4. Both throw with `.reason` populated from `detail.reason`.
5. Both attach `.pattern_class` on `scrubber_blocked`.

Documented (NOT enforced — these are intentional API differences per CLAUDE.md URL config precedence resolved at different wire-up layers):
- tools-side reads `EMMY_SERVE_URL` when `baseUrl` unset; ux-side requires explicit `baseUrl`.
- tools-side sends `timeout_ms` in the JSON body; ux-side never does (slash command relies on sidecar default + 120s outer abort).

A future drift in invariants (1)-(5) lands as a failing assertion here. The full shared-package extraction can be a follow-up phase if/when a third client appears.
