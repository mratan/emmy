# Phase 04.6: Code Review Report

**Reviewed:** 2026-04-30
**Depth:** standard
**Files Reviewed:** 16
**Status:** issues_found

## Summary

Phase 04.6 introduces Emmy's first authorized cloud-side inference bridge — a sidecar-mediated `/ask-claude` endpoint that subprocess-spawns the local `claude --print` CLI. The implementation is generally well-structured: argv invocation is genuinely shell-free (T-05 mitigated), the scrubber's reject-don't-redact path wires through correctly, the additive Pydantic schema is byte-stable per D-04, and the rate-limit logic uses an `asyncio.Lock` correctly so the in-flight + min-gap + hourly checks are atomic.

However, the review found **one Critical issue** that directly violates D-08 (privacy-by-default audit log): when the scrubber detects a secret, the matched substring (up to 40 chars of the literal secret) is written verbatim into the JSONL audit log AND returned in the HTTP error body. This defeats the entire purpose of the sha256-fingerprinting policy for the very inputs the scrubber catches.

**Warnings** cluster around three areas:
1. **Scrubber pattern coverage gaps (T-01)** — Stripe keys, AWS secret access keys, modern GitHub PATs (`github_pat_*`), Azure storage keys, generic high-entropy tokens, and DB connection strings (`postgres://user:pass@host`) are all not detected. The policy explicitly states "false negatives are NOT tolerable" but the shipping pattern set is materially incomplete.
2. **Subprocess hygiene** — on timeout, `proc.kill()` is called but `proc.wait()` is never awaited, leaving zombie subprocesses on long-lived sidecar processes.
3. **Audit-trail tampering surface (T-04)** — env-gate denials (the "default-OFF" hot path) emit zero audit entries, masking attacker toggling of `EMMY_ASK_CLAUDE`. The `append_jsonl_atomic` writer is `O_APPEND` + fsync but uses no flock, so concurrent processes (sidecar restarts during in-flight requests) could interleave partial lines on platforms where the kernel's atomic-append guarantee depends on `PIPE_BUF` for the line size.

**Info** items cover dead code, redundant defensive guards, and minor inconsistencies between the two TS HTTP clients (`packages/emmy-tools/src/ask-claude.ts` and `packages/emmy-ux/src/ask-claude-client.ts`) that could drift over time.

The Pydantic schema additive guarantee (D-04 BYTE-STABLE) is validated correctly: `AskClaudeConfig` has `enabled=False` default, all fields have non-None defaults at the *field* level but the parent `tools.ask_claude: Optional[AskClaudeConfig] = None` block default-resolves to None when absent — which is what makes the change strictly additive. Test `test_shipping_bundle_hash_unchanged_after_additive_schema` enforces this on the 7 shipping bundles. **No D-04 violation.**

## Critical Issues

### CR-01: Scrubber matched_excerpt leaks the literal secret into JSONL audit + HTTP error body (D-08 violation)

**File:** `emmy_serve/swap/controller.py:920-935` and `emmy_serve/tools/scrubber.py:73-79`

**Issue:** When the scrubber detects a secret, `scrub_result.matched_excerpt` carries the first 40 chars of the matched substring — i.e. the literal secret content (e.g. `AKIAIOSFODNN7EXAMPLE` for an AWS access key, the full `sk-ant-abc12345abcdef1234567890` for an Anthropic key). The handler then:

1. Passes `matched_excerpt` into `emit_event_ask_claude("scrubber_blocked", ...)` as a `**extras` kwarg, which the audit emitter writes verbatim into the JSONL event (line 866-867 in `emit_event_ask_claude`).
2. Returns it verbatim in the `HTTPException(400, detail={...})` body, which both the harness tool and slash command surface to the operator.

This violates D-08 ("events carry sha256 fingerprints + lengths + timing, NOT the raw prompt or response. Full content is logged ONLY when EMMY_LOG_FULL=on") for the most sensitive subset of inputs — the very ones the scrubber identified as containing secrets. The default privacy posture is silently subverted: an operator who never sets `EMMY_LOG_FULL=on` still has secrets accumulating in `~/.local/state/emmy/sidecar/ask_claude_events.jsonl` whenever the scrubber fires.

The test `test_endpoint_emits_audit_event_request` checks the *happy path* never leaks raw content, but there is no test for the scrubber-blocked path content. `test_endpoint_blocks_dirty_prompt` asserts the *audit event exists* but does not assert that the AKIA literal is absent from the event payload — the leak slips through.

The slash-command test at `slash-commands.ask-claude.test.ts:237` actually demonstrates the leak: the test prompt is `"AKIAIOSFODNN7EXAMPLE"` and the expectation is that `notifyCalls[0]![1]` matches `/aws_access_key_id/` — but in production the same path also surfaces `matched_excerpt` containing the literal AKIA key into the operator's TUI buffer.

**Fix:**

The `matched_excerpt` is useful for operator diagnostics ("which substring tripped the rule?"), but it must NOT be the literal matched text. Options, in order of preference:

```python
# Option A (preferred): hash-only — no excerpt at all in the default path.
# scrubber.py:
if m:
    return ScrubResult(
        clean=False,
        matched_class=cls_name,
        # matched_excerpt removed — class name + sha256 of the match is enough
        matched_sha256=hashlib.sha256(m.group(0).encode()).hexdigest()[:12],
    )

# controller.py — emit_event_ask_claude + HTTP detail body:
emit_event_ask_claude(
    "scrubber_blocked",
    req.prompt,
    pattern_class=scrub_result.matched_class,
    matched_sha256=scrub_result.matched_sha256,  # 12-char hash, never the secret
)
raise HTTPException(
    status_code=400,
    detail={
        "reason": "scrubber_blocked",
        "pattern_class": scrub_result.matched_class,
        # Operator-facing message tells them WHICH RULE matched, not the value
        "hint": f"matched pattern_class '{scrub_result.matched_class}'; remove the matched content and retry",
    },
)
```

```python
# Option B: gate excerpt behind EMMY_LOG_FULL=on, mirroring D-08's existing
# raw-content policy. Default path emits matched_class only.
if not scrub_result.clean:
    extras = {"pattern_class": scrub_result.matched_class}
    if os.environ.get("EMMY_LOG_FULL") == "on":
        extras["matched_excerpt"] = scrub_result.matched_excerpt
    emit_event_ask_claude("scrubber_blocked", req.prompt, **extras)
    raise HTTPException(
        status_code=400,
        detail={"reason": "scrubber_blocked", "pattern_class": scrub_result.matched_class},
    )
```

```python
# Option C (minimum): preserve matched_excerpt for the HTTP body (operator
# sees what tripped the rule), but redact in the JSONL audit. The HTTP body
# is ephemeral; the JSONL is the persistent privacy surface.
# In emit_event_ask_claude — never write matched_excerpt extras to disk:
SCRUBBER_PRIVATE_KEYS = {"matched_excerpt"}
for k, v in extras.items():
    if k not in event and k not in SCRUBBER_PRIVATE_KEYS:
        event[k] = v
```

Add a regression test:

```python
def test_scrubber_blocked_event_does_not_leak_secret(client, mock_claude, jsonl_dir, monkeypatch):
    monkeypatch.setenv("EMMY_ASK_CLAUDE", "on")
    monkeypatch.delenv("EMMY_LOG_FULL", raising=False)
    aws_key = "AKIAIOSFODNN7EXAMPLE"
    r = client.post("/ask-claude", json={"prompt": f"use {aws_key}"})
    assert r.status_code == 400
    events = _read_events(jsonl_dir)
    for ev in events:
        # The literal secret must NEVER appear in any event field.
        for v in ev.values():
            if isinstance(v, str):
                assert aws_key not in v, f"AWS key literal leaked into audit: {ev}"
```

This is a **release-blocker** for Phase 04.6: the audit log accumulates plaintext secrets every time the scrubber fires, exactly opposite to the stated D-08 invariant.

## Warnings

### WR-01: Scrubber pattern set has material false-negative gaps (T-01)

**File:** `emmy_serve/tools/scrubber_config.yaml`

**Issue:** The 11 default patterns explicitly state "False positives are tolerable; false negatives are NOT (T-01 data exfil)" but the shipping set misses widely-used secret shapes. Concrete gaps observed in 2026:

- **AWS Secret Access Key** (40-char base64 mixed): `AKIA*` access key IDs are matched, but the *secret* component (`wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY` shape) is not. Operators commonly paste both together.
- **Stripe live keys** (`sk_live_*`, `rk_live_*`, `whsec_*`, `pk_live_*`): the `sk_prefixed_key` regex requires a dash (`sk-...`) but Stripe uses underscore (`sk_live_...`). A `sk_live_4eC39HqLyjWDarjtT1zdp7dc...` would slip through.
- **Modern GitHub fine-grained PATs** (`github_pat_11ABCDEF...`): the `github_token` regex requires `gh[pousr]_` but GitHub's Aug-2022+ format uses `github_pat_*`. Common in 2026 codebases.
- **npm tokens** (`npm_*`): not covered.
- **Azure storage account keys** (88-char base64 trailing `==`): not covered.
- **Database connection strings** (`postgres://user:password@host`, `mysql://user:pass@host`, `mongodb://...`): not covered. These are extremely common in copy-pasted "help me debug" prompts.
- **Generic high-entropy strings**: a 32+ char string of mixed alphanumerics with `\b` boundaries and Shannon entropy >4.5 would catch unknown formats. Not implemented.
- **OpenAI project keys** (`sk-proj-*`): the existing `sk-(?:ant-)?` regex covers `sk-` and `sk-ant-` but not `sk-proj-`. The trailing `[a-zA-Z0-9_\-]{20,}` *might* cover it accidentally if `-proj-...` chars match — actually it would match because `-` is in the char class. So `sk-proj-XXX` works. (Verified: this one is OK.)

**Fix:** Extend `scrubber_config.yaml`:

```yaml
patterns:
  # ... existing ...
  aws_secret_access_key: '\b[A-Za-z0-9+/]{40}\b'  # high false-positive rate, accept it (D-06 policy)
  stripe_key: '\b(sk|rk|pk)_(live|test)_[a-zA-Z0-9]{24,}\b'
  stripe_webhook: '\bwhsec_[a-zA-Z0-9]{32,}\b'
  github_pat_finegrained: '\bgithub_pat_[A-Z0-9]{22}_[a-zA-Z0-9]{59}\b'
  npm_token: '\bnpm_[a-zA-Z0-9]{36}\b'
  azure_storage_key: '\b[A-Za-z0-9+/]{86}==\b'
  db_url_password: '\b(postgres(?:ql)?|mysql|mongodb(?:\+srv)?|redis)://[^\s:@/]+:[^\s@/]+@'
  # Generic high-entropy fallback (last; tolerates false positives per D-06):
  long_opaque_token: '\b[A-Za-z0-9_\-]{40,}\b'  # CAUTION: very broad — gate on env or remove
```

The `long_opaque_token` line will produce false positives on long file paths or hashes; consider gating it behind an `EMMY_SCRUBBER_AGGRESSIVE=on` opt-in. The first 7 additions are all low-FP and should be unconditional.

Add tests in `tests/unit/test_scrubber.py`:

```python
@pytest.mark.parametrize("prompt,expected_class", [
    ("STRIPE_KEY=sk_live_4eC39HqLyjWDarjtT1zdp7dcThisIsTheSecret", "stripe_key"),
    ("token github_pat_11ABCDEFGHIJKLMNOPQRST_abcdefghij...", "github_pat_finegrained"),
    ("DATABASE_URL=postgres://user:hunter2@db.example.com:5432/app", "db_url_password"),
    ("npm config token npm_abcdefghijklmnopqrstuvwxyz0123456789", "npm_token"),
])
def test_scrubber_extended_coverage(prompt, expected_class):
    r = scrub(prompt)
    assert r.clean is False
    assert r.matched_class == expected_class
```

### WR-02: Subprocess timeout leaks zombie processes (no `await proc.wait()` after kill)

**File:** `emmy_serve/swap/controller.py:988-1002`

**Issue:** On `asyncio.TimeoutError`, the handler calls `proc.kill()` (synchronous SIGKILL delivery) but never awaits `proc.wait()` afterward. asyncio's subprocess transport keeps the child PID and pipe FDs alive until `wait()` is called. Under sustained timeout pressure (e.g. `claude` hangs because Anthropic's API is slow, operator submits 30 prompts that all hit the 60s default timeout), the sidecar accumulates zombie PIDs — each holding 3 FDs (stdin/stdout/stderr pipes). At Linux's default `RLIMIT_NOFILE=1024` per process, the sidecar will eventually fail to spawn new subprocesses with "too many open files."

The audit event captures `duration_ms` correctly because the timeout fires BEFORE the kill, so timing is preserved. The issue is purely resource-cleanup.

**Fix:**

```python
except asyncio.TimeoutError:
    duration_ms = int((time.monotonic() - t0) * 1000)
    emit_event_ask_claude(
        "error", req.prompt,
        duration_ms=duration_ms,
        reason="timeout",
    )
    try:
        proc.kill()
    except Exception:
        pass
    # Reap the zombie so its FDs are released. Bounded wait so a
    # stuck child (rare with SIGKILL) doesn't hang the handler.
    try:
        await asyncio.wait_for(proc.wait(), timeout=2.0)
    except (asyncio.TimeoutError, Exception):
        # Pathological — child ignored SIGKILL or transport already cleaned.
        # The leaked zombie is rare and bounded by the 2s wait we just did.
        pass
    raise HTTPException(
        status_code=504,
        detail={"reason": "timeout"},
    )
```

Add a regression test that drives a timeout and then asserts the spy's `kill_called` is True AND the FakeProc's `wait()` was awaited (extend `_FakeProc` with an `awaited_wait: bool` flag).

### WR-03: Env-gate denials emit zero audit entries (T-04 audit-trail tampering)

**File:** `emmy_serve/swap/controller.py:903-908`

**Issue:** The handler intentionally short-circuits with `503 env_disabled` BEFORE any audit emission (per the inline comment "no subprocess spawn, no audit trail (env-off is the air-gap default; logging would itself be an unwanted side-effect)"). This is reasonable for the *intended* default-off use case — but it also means that if an attacker (or a buggy systemd unit, or a confused operator) toggles `EMMY_ASK_CLAUDE=on` briefly to sneak through some prompts and then toggles it off, no record exists of the prompts that were rejected during the off-window.

More importantly, the *direction* of an env-gate flap is invisible to forensics: an operator looking at the audit log sees only successful + scrubber-blocked + rate-limited events, with no signal about how often `EMMY_ASK_CLAUDE` was toggled or how many requests bounced off it. T-04 (audit-trail tampering) lists exactly this scenario: an attacker can probe the env-gate state cheaply (`POST /ask-claude` returns 503 vs other codes) and flip the env to land prompts during a window with zero forensic trace.

**Fix:** Emit a minimal `env_gate_denied` event on the env-disabled path. The event needs to carry the prompt sha256 + length, NOT the prompt content (the comment's concern about side-effects-when-off is valid for full-content logging, but the sha256+length is privacy-preserving by design).

```python
if os.environ.get("EMMY_ASK_CLAUDE") != "on":
    # T-04 — record the denial so an env flap is observable in audit.
    # sha256+length only; the EMMY_LOG_FULL gate still suppresses raw content.
    emit_event_ask_claude("env_gate_denied", req.prompt)
    raise HTTPException(
        status_code=503,
        detail={"reason": "env_disabled"},
    )
```

This adds N+1 lines per probe but the volume is bounded by the operator's request rate; real abuse will surface as a spike in `env_gate_denied` events.

### WR-04: append_jsonl_atomic uses no advisory lock — concurrent writers can interleave

**File:** `emmy_serve/diagnostics/atomic.py:62-75`

**Issue:** The function opens the file in `O_APPEND` mode and relies on Linux's atomic-append guarantee, which holds only for writes ≤ `PIPE_BUF` (typically 4096 bytes). The `ask_claude` audit events are usually small (sha256 + lengths + timing ≈ 200-400 bytes) so this is fine in practice, BUT:

1. When `EMMY_LOG_FULL=on` is set, events carry the full prompt + response, which can easily exceed 4 KiB (the AskClaudeRequest cap is 200 KiB).
2. In a sidecar restart-during-in-flight scenario (rare but possible: `systemctl restart emmy-sidecar` mid-request), a new sidecar process can start writing to the same JSONL file before the old process's `with open` block exits.

Result: lines from concurrent writers can interleave, producing JSONL parse errors downstream (the `_read_events` test helper would silently drop malformed lines).

**Fix:** Use `fcntl.flock(LOCK_EX)` around the write. Linux's flock is advisory but every emmy writer goes through `append_jsonl_atomic`, so the convention is enforceable:

```python
import fcntl

def append_jsonl_atomic(path: str | Path, obj: dict) -> None:
    dest = Path(path)
    dest.parent.mkdir(parents=True, exist_ok=True)
    line = json.dumps(obj, sort_keys=True, separators=(",", ":")) + "\n"
    with open(dest, "a", encoding="utf-8") as f:
        fcntl.flock(f.fileno(), fcntl.LOCK_EX)
        try:
            f.write(line)
            f.flush()
            os.fsync(f.fileno())
        finally:
            fcntl.flock(f.fileno(), fcntl.LOCK_UN)
```

This change is upstream of `ask_claude` (the helper is shared with KV-finder and thermal logs) but the impact analysis is local: every existing caller gets atomic interleave-free writes, no API change.

Add test driving 50 concurrent `append_jsonl_atomic` calls from `concurrent.futures.ProcessPoolExecutor` and asserting every line parses.

### WR-05: TS HTTP client has no per-error-shape regression test asserting `matched_excerpt` is NOT logged client-side

**File:** `packages/emmy-tools/src/ask-claude.ts:354-366`

### WR-06: Two TS HTTP clients duplicate the same wire contract with subtle differences

**File:** `packages/emmy-tools/src/ask-claude.ts:191-259` and `packages/emmy-ux/src/ask-claude-client.ts:106-169`

**Issue:** Two near-identical sidecar HTTP clients exist:
- `callAskClaudeViaSidecar` in `@emmy/tools` — used by the model-side tool
- `callAskClaude` in `@emmy/ux/ask-claude-client` — used by the slash command

They have subtle behavioral differences:
- The tools-side client sends `{prompt, timeout_ms?}` in the body; the ux-side client sends only `{prompt}` — meaning the slash-command path can't override the timeout even though the sidecar supports it.
- The tools-side client has no default timeout; the ux-side client defaults to `120_000` ms.
- The tools-side client checks `process.env.EMMY_SERVE_URL` and falls back to `127.0.0.1:8003`; the ux-side client requires the caller to resolve `baseUrl` upstream and never reads `process.env`.
- The error-body parsing is structurally identical but written twice with subtly different type guards.

These will drift. A future change to the sidecar wire contract (e.g. adding a new error reason or a `request_id` field) requires updating both — and the tests don't enforce that they stay synchronized.

**Fix:** Extract a shared `@emmy/sidecar-protocol` package (or co-locate in `@emmy/telemetry/wire-contracts.ts`) that exports a single `callAskClaude({baseUrl, prompt, timeoutMs?, fetchImpl?})` and a single `AskClaudeError` interface. Both packages depend on it. The deduplication also reduces the surface area for CR-01-class issues — a single bug fix lands in one place.

Lower-priority alternative: add a contract-test that imports BOTH clients and exercises the same fake fetch scenarios, asserting identical error shapes.

### IN-02: Boilerplate-tried regex `^\s*(i don'?t know|i tried thinking)\b` allows trivial bypass

**File:** `packages/emmy-tools/src/ask-claude.ts:129-133`

**Issue:** The boilerplate-tried gate matches prefixes like "I tried thinking" anchored on `^` only (no `$`). This is intentional per the inline comment ("I tried thinking through approach A and it returned X" should pass). But the gate also accepts:

- "I tried thinking. Then I tried thinking again." → blocked (good)
- "Honestly I tried thinking" → passes (the `^` anchor requires it to be the FIRST text)
- "I 'tried thinking'" → blocked (matches via `i tried thinking` anywhere... actually no, anchored ^)

A model that wants to bypass the gate just prepends a single character: "x I tried thinking about it" passes. This is a discipline-shaper, not a security gate, so this is acceptable — but it's worth documenting that the gate is intentionally easy to bypass and that the *real* enforcement is the rate-limit + the model's prompt-following compliance.

**Fix:** Add a comment to the regex declaration:

```ts
// Discipline shaper, NOT a security gate. A determined model can bypass with
// a leading character. Real enforcement is the per-turn rate-limit + the
// description's "must show in your reasoning trace" prompting (D-15).
const _BOILERPLATE_TRIED_PATTERNS: readonly RegExp[] = [...]
```

### IN-03: `_record_error` global write race is technically possible but practically benign

**File:** `emmy_serve/swap/controller.py:214-217`

**Issue:** `_record_error` mutates the module-level `_last_error` dict without holding any lock. Two concurrent handlers (e.g. `/start` and `/ask-claude` failing simultaneously) could both write — Python's GIL guarantees the dict assignment itself is atomic, but a subsequent `GET /status` reading `_last_error` could see fields from either error. In practice, GETting /status while two errors fire concurrently is rare and the worst case is a single inconsistent status response. Worth documenting.

**Fix:** Either guard with an `asyncio.Lock` (overkill given the rarity), or document the eventual consistency:

```python
def _record_error(msg: str) -> None:
    """Update the module-level _last_error dict; surfaced via GET /status.
    
    Note: not lock-guarded. Concurrent errors can race — the dict assignment
    itself is GIL-atomic but a GET /status reading mid-race can observe a
    partial state. Acceptable for this single-user single-process sidecar.
    """
    global _last_error
    _last_error = {"ts": _now_iso(), "msg": msg}
```

### IN-04: `secret_path_keyfile` regex matches arbitrary `.pem` strings — high false-positive on docs

**File:** `emmy_serve/tools/scrubber_config.yaml:19`

**Issue:** `'[a-zA-Z0-9_-]+\.(pem|key|p12|pfx|jks)\b'` matches any filename with these extensions, including innocuous strings like "see foo.key in the docs" or a discussion of "openssl x509 -in cert.pem". The D-06 policy is "false positives are tolerable" so this is fine, but it interacts with CR-01: when the FP fires, the JSONL audit logs the matched filename verbatim, which on a private codebase could itself leak project naming patterns.

**Fix:** Once CR-01 is fixed (no plaintext excerpt in audit), this becomes a pure FP-rate concern. Consider tightening to require a path-shaped prefix:

```yaml
secret_path_keyfile: '(?:^|[\s/])[a-zA-Z0-9_/.-]+\.(pem|key|p12|pfx|jks)\b'
```
