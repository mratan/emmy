# Phase 04.7 — Mistral Medium 3.5 128B alternate (eval-only, GGUF Q4_K_M)

**Status:** SCOPED — ready to plan/execute
**Drafted:** 2026-05-01
**Phase shape:** mid-milestone insertion, fourth alternate profile (after Phase 4.1's Qwen-27B + Gemma-31B + Phase 4's Gemma-26B-A4B)

## Purpose

Add a **128B-class dense alternate** profile to the Spark stack so Phase 5 eval can include a heavyweight quality data point alongside the existing 4B-active and 27-31B-dense participants. Profile is opt-in via `/profile mistral-medium-3.5`, **not** a daily-driver candidate, **not** a routes.yaml participant. The thesis: at ~3 tok/s decode it's not viable for 8h/day coding sessions, but the SWE-bench Verified 77.6% / τ³-Telecom 91.4% quality bar makes it worth carrying for occasional heavy-reasoning escalations and as a calibration data point in the dense-vs-MoE Phase 5 matrix.

## Background

- HF release: `mistralai/Mistral-Medium-3.5-128B` (BF16, ~200 GiB on disk). Dense 128B, 88 decoder layers, GQA 12:1 (96 Q heads / 8 KV heads, head_dim 128), 256K native context (`max_position_embeddings=262144`, no sliding window).
- Three quant providers surveyed on 2026-05-01:
  - `RecViking/Mistral-Medium-3.5-128B-NVFP4` — 74.8 GB, vLLM-nightly compatible, but Emmy's prior research (`CLAUDE.md` Pinned Tech Stack) shows NVFP4 is **slower than FP16 on GB10 UMA** (-23.6% at 32K context). Built for 4× RTX 5090 TP=4; Spark would need single-GPU re-tune.
  - `bartowski/mistralai_Mistral-Medium-3.5-128B-GGUF` — 22 quant variants from IQ1_S to Q8_0; Q4_K_M at 73.0 GB is the community-gold-standard 4-bit balance.
  - `unsloth/Mistral-Medium-3.5-128B-GGUF` — empty repo as of 2026-05-01; Mistral flagged GGUF support as WIP for long-context.
- Throughput math (GB10 LPDDR5X ~273 GB/s, dense decode bandwidth-bound):
  | Profile | Pool memory | Theoretical max | Realistic |
  |---|---|---|---|
  | Gemma 4 26B-A4B FP8 (daily-driver) | 4 GB active | 68 tok/s | ~36 (measured) |
  | Qwen 27B dense FP8 | 27 GB | 10 | ~7.6 (measured) |
  | Gemma 31B dense FP8 | 31 GB | 8.8 | ~6.4 (measured) |
  | **Mistral 128B Q4_K_M** | **73 GB** | **3.7** | **~2.5–3 tok/s estimated** |
- KV math: `2 × 88 layers × 8 KV heads × 128 head_dim × 1 byte (FP8)` = **176 KB/token**. At 128K context = **22 GB**.
- Container choice: existing pinned containers don't ship vLLM's GGUF backend (NGC `26.03.post1-py3` is fastsafetensors-derived; upstream Day-1 `gemma4-0409-arm64-cu130` is BF16-FP8). New pin: `vllm/vllm-openai:nightly-arm64-cu130` (digest captured at first boot smoke).
- See discussion thread in this session's transcript (operator + claude, 2026-05-01) for the full architectural deliberation, including the rejected NVFP4 path and the operator's three guidance points (skip KV-bisect, skip thermal, full 128K context).

## Locked Decisions

| ID | Decision | Rationale |
|----|---|---|
| **D-01** | **Q4_K_M GGUF from bartowski**, not NVFP4. | Community-gold-standard 4-bit; multi-part split totaling 73.0 GB; same memory footprint as NVFP4 without the GB10-UMA slowdown. NVFP4's 4-bit-tensor-core advantage on Blackwell dGPU does not translate to GB10's UMA topology per Emmy's measured `-23.6% vs FP16` finding. |
| **D-02** | **vLLM with `--quantization gguf` (experimental backend)**, single-GPU. | Preserves Emmy's vLLM-only architecture (existing OpenAI-compat surface, prefix caching, sidecar `/profile` swap, tool-call parser, audit JSONL). Avoids introducing llama.cpp as a parallel runtime. Acknowledged trade-off: GGUF backend is experimental in vLLM; long-context is flagged WIP by Mistral. Boot smoke validates basic shape. |
| **D-03** | **`max_model_len: 131072` (128K)** — half of native 256K. | At 256K + Q4_K_M weights (73 GB) + FP8 KV (44 GB) we'd hit 117/128 GB UMA leaving ~11 GB for system+sidecar — too tight for normal Spark coexistence (bio pipeline / harness CPU / MCP servers). 128K KV at FP8 = 22 GB → total 95 GB, leaves 28+ GB for the rest of the box. Operator-shareable. The 256K capability is documented as a future option requiring "Spark-exclusive while loaded" operator gesture; not pursued in v1. |
| **D-04** | **`gpu_memory_utilization: 0.78`** — operational, not RAM-headroom retune. | Pool budget = 0.78 × 128 GB = 99.8 GB. Sized to fit weights (73) + KV (22) + ~5 GB scratch. Departs from the recent dense-profile convention of 0.55 because at 0.55 the pool (70 GB) is below weight size — the model wouldn't load. Higher gmu is structurally required, not a risk choice. System headroom (~28 GB) is the operator-shareability budget. |
| **D-05** | **Skip formal KV-bisection protocol.** Substitute: single boot smoke at gmu=0.78 + 32K prefill probe; step down 0.05 (0.78 → 0.73 → 0.68) if OOM. | Bisection's audit value (find the box ceiling) is well-known: 4 prior profiles all converge to 0.86 hardware ceiling. Operational ceiling is determined by structural fit (D-04), not bisect. The failure mode at gmu=0.78 is loud (OOM at boot or first long prefill), not silent. ~30 min vs 2-4h. PROFILE_NOTES.md records "operational ceiling = 0.78, validated by single-run smoke" — honest about the protocol used. |
| **D-06** | **Skip 2× 2h thermal replay protocol.** | Empirical record on Spark: 8 hours of cumulative thermal-replay data across 4 prior profiles, **zero throttle observed**. The original "2.8→2 GHz throttle" claim originates from `PITFALLS.md` line 815 citing an NVIDIA Developer Forum report — never measured on this hardware. For Mistral 128B Q4_K_M specifically, the thermal envelope is **lighter** than the dense profiles that already passed (decode ~3 tok/s vs 7-8, less sustained heat, occasional-use not 8h/day). Pitfall #4 status updated to "characterized on Spark across 4 profiles, no throttle observed; gate retained for daily-driver candidates, skipped for non-daily-driver alternates." |
| **D-07** | **`max_num_seqs: 1` — single-stream.** | At 73 GB weights + 22 GB KV the pool is tight; allowing concurrent streams would multiply KV by sequence count and OOM. Single-stream is the operational shape for an occasional-escalation profile anyway (no batching value when used 1-5 times per session). |
| **D-08** | **`kv_cache_dtype: fp8`** — required, not optional. | At BF16 KV cache, 128K context = 44 GB; total 73 + 44 = 117 GB → too tight at gmu=0.78 (pool capped at 99.8 GB). FP8 KV halves it to 22 GB making 128K achievable. Quality cost is small for a non-daily-driver profile; eval matrix can flip BF16 KV at smaller context if calibration deems it necessary. |
| **D-09** | **`tool_call_parser: mistral` + `enable_auto_tool_choice: true`** — match Mistral 3.x conventions. | The model is RL-trained for Mistral-style tool calls (per the upstream model card's "powers Mistral Vibe agent" line). Mirroring the parser ensures Emmy's harness sees structured tool calls, not raw text. Same discipline used by Qwen-27B (qwen3-coder parser) and Gemma-26B-A4B (default parser). |
| **D-10** | **NEW container pin: `vllm/vllm-openai:nightly-arm64-cu130`** with digest captured at first boot smoke. | Existing pinned containers don't ship the GGUF backend on the version we need. Container-per-family rule (per Phase 4 closeout) extends to: Mistral GGUF lives on its own pin. Digest is captured during plan 04.7-02 boot smoke and locked into `serving.yaml`'s `container_image_digest`. Pitfall #2 (vLLM API churn) accepted as a re-pin debt for nightly; documented in PROFILE_NOTES.md. |
| **D-11** | **Single profile version: `v1` only.** No sibling-of-siblings (no `v1`/`v1.1` audit/operational split). | The dense-profile sibling pattern (v1 = KV-ceiling audit @ 0.86, v1.1 = operational @ 0.55) is a Phase 4.1 invention for daily-driver-grade discipline. Mistral 128B is eval-only; the operational profile IS the audit profile. Future tweaks (e.g., context bump, gmu retune) can cut a v2 if needed. |
| **D-12** | **NOT a `routes.yaml` participant; NOT a daily-driver candidate; NOT a `tools.ask_claude.enabled` flip target.** | Eval-only opt-in via `/profile mistral-medium-3.5`. Daily-driver stays Gemma 4 26B-A4B v2.1. Within-model role routing (Phase 4 SC-3) doesn't apply (no plan/edit/critic siblings). The escalation tool form stays `ask_claude` for now; whether a future `ask_mistral` local-escalation tool gets built depends on Phase 5 quality vs. throughput findings. |
| **D-13** | **Eval-only profile.** Phase 5 evaluates this against the existing 3 profiles in the dense-vs-MoE matrix. Throughput is informational only — gate is **bootability + first chat completion + tool call**. | Mirrors Phase 4.1's eval-only directive: "throughput informational only, gate is thermal stability." Here we drop thermal stability too (D-06); the gate is just "does it boot and answer?" |

## Threat Model

| ID | Threat | Mitigation |
|----|---|---|
| **T-01** | **GGUF backend instability.** vLLM's GGUF loader is experimental; might fail to bootstrap, fail at long context, or produce nonsense outputs. | Boot smoke at 32K first, then step up to 128K. If long-context degrades (Mistral's flagged WIP issue), document a `max_model_len` ceiling in PROFILE_NOTES.md. Eval-only profile means no daily-driver impact if it fails. |
| **T-02** | **Container nightly drift.** vLLM nightly tag advances; old digest may pull invalid manifest within weeks. | Pin specific digest at boot smoke (Plan 04.7-02). PROFILE_NOTES.md documents "re-pin every ~4 weeks if profile is to remain bootable." Acceptable for an eval-only profile; not acceptable for daily-driver. |
| **T-03** | **OOM at gmu=0.78 on Spark.** GGUF backend may have higher memory overhead than fastsafetensors, exceeding the 99.8 GB pool. | Boot smoke detects this loudly. Step-down protocol: 0.78 → 0.73 → 0.68 (if 0.68 fails, drop max_model_len from 131072 to 65536). PROFILE_NOTES.md records the converged value. |
| **T-04** | **Tool-call parser mismatch.** Mistral 3.5's tool-call format may differ subtly from the parser vLLM ships. | Boot smoke includes a tool-call probe (canary prompt that asks for a structured tool call). Failure → fall back to default parser + document in PROFILE_NOTES.md. Same canary pattern as Phase 4.1's Qwen 27B mid-stream fix. |
| **T-05** | **Long-context generation degradation.** Mistral flagged GGUF long-context as WIP. | Boot smoke generation test at 32K context (well below 128K). 128K listed as `max_model_len` but real-world use likely <16K for first-pass. If 128K turns out unusable, re-cap to 32K via v2 cut. |
| **T-06** | **Spark coexistence at gmu=0.78.** ~28 GB system headroom may not be enough for bio pipeline + sidecar + harness during a /ask-mistral call. | Operator-monitoring during first real-use sessions. If sidecar OOMs again (cf. Phase 04.2 OOM postmortem), drop max_model_len to 96K (gmu pool ~88 GB → ~40 GB system headroom). |

## Deferred / Out of Scope

- **256K full context.** Possible only with operator-exclusive Spark mode (no other workloads); not pursued in v1. Future v2 cut if demand justifies.
- **NVFP4 alternate.** RecViking's NVFP4 build is the obvious comparison point; Phase 5 may add it as an A/B if measurement is cheap. Not built in v1 because GGUF Q4_K_M is the higher-confidence path.
- **`ask_mistral` local-escalation tool.** A local-only sibling of Phase 04.6's `ask_claude` (no cloud egress). Worth considering if Mistral 128B turns out high-quality enough that operator wants it as the "ask hard reasoning" path. Defer until Phase 5 quality data is in.
- **Phase 4 SC-3 within-model role routing.** No plan/edit/critic siblings shipped; routes.yaml stays Gemma-only. Ship after eval shows Mistral is worth the effort.
- **Promotion to daily-driver candidate.** Throughput math says no (~3 tok/s), but if Phase 5 calibration surprises us, we'd revisit. Decision deferred to Phase 5 closeout.
- **2× 2h thermal replay.** Skipped per D-06; if Pitfall #4 ever fires on this Spark in real use, Phase 4.1's protocol can be re-applied retroactively.
- **Formal KV-bisection.** Skipped per D-05; can be run retroactively if a future profile is promoted to daily-driver candidacy.

## Acceptance Gates

The phase closes when **all** of the following pass:

1. **Bootable** — `bash scripts/start_emmy.sh --profile profiles/mistral-medium-3.5/v1` returns `emmy-serve ready` with vLLM `/v1/models` reporting `mistral-medium-3.5`.
2. **First chat completion green** — `pi-emmy --print "Reply with: SP_OK_TEST_PASS"` returns the exact token; verifies system-prompt delivery (Pitfall #6) at 128K context.
3. **Tool-call probe green** — canary prompt produces a structured Mistral-format tool call parsed by vLLM into the OpenAI tool-call shape. (Same canary discipline used by Qwen-27B mid-stream fix in Phase 4.1.)
4. **`/profile mistral-medium-3.5` swap green** — atomic 4-phase swap from daily-driver Gemma → Mistral and back works end-to-end via the Phase 4 D-23 path.
5. **Profile validate green** — `uv run emmy profile validate profiles/mistral-medium-3.5/v1` exits 0; bundle hash captured in PROFILE_NOTES.md.
6. **Documentation in tree** — CLAUDE.md, README.md, eval/MATRIX.md, runbook all updated and committed.
7. **No regressions** — full Python pytest + emmy-ux/emmy-tools bun test green at close (8 pre-existing emmy-ux fails OK; non-regressive).

**Operator-time validation** replaces the formal thermal/eval gates: real-use observation over the next ~week of /profile mistral-medium-3.5 invocations is the implicit acceptance signal. If sidecar OOMs, generation degrades, or memory pressure cascades into bio-pipeline territory, the gmu/max_model_len knobs get retuned per T-03/T-06.
