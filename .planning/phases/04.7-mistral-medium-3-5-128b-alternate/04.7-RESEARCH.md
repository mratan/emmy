## Summary (5 bullets — what the planner most needs)

1. **Schema extension is unavoidable.** `engine.quantization` is currently `Literal["fp8", "bf16", "auto"]` (`emmy_serve/profile/schema.py:83`). Mistral GGUF requires `"gguf"`. Planner MUST add a Plan 04.7-01 task that extends the Literal, OR cuts the Mistral profile under a different mechanism. The schema is `extra="forbid", frozen=True`. Without this change, `emmy profile validate` rejects the bundle and G-5 cannot pass.

2. **CLI renderer has two gaps that bite Mistral specifically.** `render_vllm_cli_args()` (`emmy_serve/boot/runner.py:53–98`) currently emits NEITHER `--reasoning-parser` NOR `--max-num-seqs` NOR `--tokenizer`. Mistral's HF model card explicitly recommends `--reasoning-parser mistral`. Mistral 128B Q4_K_M GGUF specifically REQUIRES `--tokenizer mistralai/Mistral-Medium-3.5-128B` (vLLM GGUF docs: tokenizer extraction from GGUF is "time-consuming and unstable"). Planner MUST extend the renderer in Plan 04.7-01. Note: existing Gemma profiles set `reasoning_parser: gemma4` in serving.yaml but it never reaches the vLLM CLI today — that's either a latent Gemma bug or vLLM auto-detects from tokenizer_config.json. This phase forces the issue to surface.

3. **Multi-part GGUF: vLLM ONLY supports single-file.** Confirmed via vLLM official docs (https://docs.vllm.ai/en/stable/features/quantization/gguf/): *"Currently, vllm only supports loading single-file GGUF models."* The bartowski Q4_K_M release IS split (78.4 GB total, "Split: true" in metadata). Plan 04.7-02 MUST include a `gguf-split --merge` step using llama.cpp's bundled tool. **CONTEXT step `concatenate first` is wrong** — `cat parts > out.gguf` does NOT produce a valid merged GGUF. Use `gguf-split --merge <first-shard.gguf> <output.gguf>`; tool auto-detects shards 2..N from the first.

4. **Container tag in CONTEXT may be inverted.** CONTEXT D-10 names `vllm/vllm-openai:nightly-arm64-cu130`; Docker Hub evidence (cross-verified via WebSearch + Stevescargall blog 2026-04 + vLLM PR references) shows the actual tags are `cu130-nightly` and `cu130-nightly-aarch64`. Plan 04.7-02 first task: try `docker pull vllm/vllm-openai:cu130-nightly-aarch64` (likely correct), capture `docker inspect --format '{{.Id}}'` digest, write to `serving.yaml`. If the literal CONTEXT spelling resolves, use it as-is (Docker Hub may host an alias). Either way the digest is the source of truth (`render_image_ref` uses bare-digest form for `vllm/vllm-openai` repo per `runner.py:48–50`).

5. **Plan 04.7-02 boot smoke leverages the existing 4-step canary chain at no extra code cost** — `scripts/smoke_test.py` already invokes `wait_for_vllm` → `run_sp_ok` → `run_tool_call` → `run_generate`. This satisfies G-1, G-2, G-3 in one shot for any profile bundle with a valid `serving.yaml`. The only Mistral-specific work is choosing whether to override `tool_call.py:TOOL_CALL_USER_MESSAGE` for Mistral's `[TOOL_CALLS]` format — likely NOT needed because the parser converts to OpenAI shape (the canary asserts `tool_calls[0].function.name == "read_file"`, format-agnostic).

**Primary recommendation:** Plan 04.7-01 is heavier than CONTEXT implies. It must (a) extend pydantic Literal for `quantization: gguf`, (b) extend `render_vllm_cli_args` to emit `--reasoning-parser`, `--max-num-seqs`, `--tokenizer`, (c) author the profile bundle. Plan 04.7-02 is straightforward IF (a) and (b) land. Plan 04.7-03 is documentation-only.

### 1.3 Multi-file GGUF assembly (Plan 04.7-02 step)

CONTEXT in-scope step 3 says "concatenated to `/models/Mistral-Medium-3.5-128B-Q4_K_M.gguf`". **Plain `cat` is WRONG for GGUF.** Use `gguf-split` from llama.cpp:

```bash
# Step 1 — download all parts (HF CLI handles glob)
huggingface-cli download bartowski/mistralai_Mistral-Medium-3.5-128B-GGUF \
  --include "mistralai_Mistral-Medium-3.5-128B-Q4_K_M/*" \
  --local-dir /models/Mistral-Medium-3.5-128B-Q4_K_M-parts

# Step 2 — install gguf-split (one-time per host; Spark already has llama.cpp? must verify)
git clone https://github.com/ggml-org/llama.cpp /tmp/llama.cpp
cd /tmp/llama.cpp && cmake -B build && cmake --build build --config Release -j
# binary lands at /tmp/llama.cpp/build/bin/llama-gguf-split

# Step 3 — merge (point at FIRST shard; tool auto-discovers 2..N in same dir)
/tmp/llama.cpp/build/bin/llama-gguf-split --merge \
  /models/Mistral-Medium-3.5-128B-Q4_K_M-parts/mistralai_Mistral-Medium-3.5-128B-Q4_K_M-00001-of-NN.gguf \
  /models/Mistral-Medium-3.5-128B-Q4_K_M.gguf

# Step 4 — verify single-file size ≈ 73 GB (CONTEXT) or 78.4 GB (HF metadata, may be uncompressed)
ls -lh /models/Mistral-Medium-3.5-128B-Q4_K_M.gguf
# Step 5 — clean up parts dir to reclaim ~78 GB
rm -rf /models/Mistral-Medium-3.5-128B-Q4_K_M-parts
```

**Disk path convention:** existing profiles use `/models/<repo-basename>` (verified via `ls /data/models/`: `gemma-4-26B-A4B-it`, `Qwen3.6-27B-FP8`, etc. — `/data/models` is bind-mounted into the container at `/models` per `runner.py:120`). So the in-container path is `/models/Mistral-Medium-3.5-128B-Q4_K_M.gguf`; the host path is `/data/models/Mistral-Medium-3.5-128B-Q4_K_M.gguf`. Profile `engine.model` field = container-internal path.

## 3. Tool-call parser (D-09)

### 3.1 Mistral parser — output shape

Mistral models emit tool calls as `[TOOL_CALLS]` followed by JSON array:

```
[TOOL_CALLS]read_file{"path": "/etc/hosts"}
```

vLLM's `mistral` tool parser converts this to OpenAI shape:

```json
{
  "role": "assistant",
  "content": null,
  "tool_calls": [{
    "id": "call_abc",
    "type": "function",
    "function": {"name": "read_file", "arguments": "{\"path\": \"/etc/hosts\"}"}
  }]
}
```

Sources:
