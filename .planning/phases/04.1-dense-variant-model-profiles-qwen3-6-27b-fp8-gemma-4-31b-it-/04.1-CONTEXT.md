# Phase 04.1 — Dense-variant model profiles

## Goal (one sentence)

Add dense sibling profiles to both existing model families — `Qwen/Qwen3.6-27B-FP8` alongside the `qwen3.6-35b-a3b` MoE, and `google/gemma-4-31B-it` alongside the `gemma-4-26b-a4b-it` MoE — with the same full profile discipline (KV bisection, 2×2 h thermal replay, profile validate + hash) as Phase 1/4 so both are Phase 5-eval-ready.

## Why now

Phase 5 is the eval harness phase. Having dense-vs-MoE siblings in-tree before Phase 5 starts lets the eval matrix include a "dense vs MoE at same param budget" axis as a first-class comparison, instead of bolting it on after the fact. Neither dense profile changes the daily-driver default — both are opt-in via `/profile`.

## Research already done (2026-04-24 WebSearch + WebFetch)

**Qwen 3.6 dense (released 2026-04-22):**
- HF repo: `Qwen/Qwen3.6-27B-FP8` (27 B params, dense, publisher-shipped FP8)
- Context native 262,144 (YaRN-extensible to ~1M)
- Arch class `Qwen3ForCausalLM` — same as existing MoE
- Same tokenizer + chat template + tool_call_parser as `Qwen3.6-35B-A3B-FP8`
- Expected throughput on DGX Spark GB10: ~40-55 tok/s (bandwidth-bound dense vs MoE ~75)
- vLLM 0.19+ supported per official recipes page
- Apache 2.0 license

**Gemma 4 dense (released 2026-04-02):**
- HF repo: `google/gemma-4-31B-it` (30.7 B params, dense, instruction-tuned)
- 256 K context, hybrid (sliding-window + global) attention
- **No publisher FP8/NVFP4/INT4** — Google ships BF16 only (~61 GB weights at BF16)
- Community / partner quants exist (`nvidia/Gemma-4-31B-IT-NVFP4`, etc.) but NVFP4 is disqualified per STACK.md D-13 (−23.6 % at 32K context on GB10)
- vLLM 0.19+ supported, **requires** `vllm/vllm-openai:gemma4-0409-arm64-cu130` container (same one Phase 4 v2 bundle already uses — Transformers ≥5.5.0 requirement)
- Expected throughput: ~6-10 tok/s (bandwidth-bound dense, matches prior research flag in CLAUDE.md's stack note)

## Plan breakdown (recommended)

### Plan 04.1-1: `profiles/qwen3.6-27b/v1/` bundle

**Scope:** clone-and-retarget from `profiles/qwen3.6-35b-a3b/v3.1/` (the current daily-driver profile) — not v3.2 because v3.1 is the locked validated baseline. Change:
- `profile.yaml`: `id: qwen3.6-27b`, `version: v1`, provenance note
- `serving.yaml.engine.model`: `/models/Qwen3.6-27B-FP8`
- `serving.yaml.engine.model_hf_id`: `Qwen/Qwen3.6-27B-FP8`
- `serving.yaml.engine.served_model_name`: `qwen3.6-27b`
- `serving.yaml.engine.quantization`: whatever the publisher's FP8 weights dictate (likely `fp8`, verify via `config.json` from the HF repo before writing)
- Keep container: `nvcr.io/nvidia/vllm:26.03.post1-py3` (NGC, fastsafetensors supported)
- Keep tool_call_parser: whatever Qwen 3.6 family parser the NGC container ships (Qwen3-MTP era tool parser — same as existing MoE)
- `serving.yaml.engine.gpu_memory_utilization`: seed at 0.55 (Phase-3.1-style RAM-conservative seed); final value written by `scripts/find_kv_budget.py`
- Prompts / tool schemas / grammars: byte-identical clone from v3.1 (no model-specific changes expected for dense)
- Update `harness.yaml.context.max_input_tokens` derivation via `scripts/compute_max_input_tokens.ts`

**Tasks:**
1. Pull weights: `huggingface-cli download Qwen/Qwen3.6-27B-FP8 --local-dir /models/Qwen3.6-27B-FP8`
2. Pin container image digest (`docker pull` + capture `Id` via `docker inspect`)
3. Write profile bundle + recompute hash via `emmy profile hash --write`
4. `uv run emmy profile validate profiles/qwen3.6-27b/v1` exits 0
5. Smoke test: `scripts/start_emmy.sh --profile profiles/qwen3.6-27b/v1` → `smoke ok: tok/s=...`
6. KV bisection: `uv run python scripts/find_kv_budget.py --profile profiles/qwen3.6-27b/v1 --drive-minutes 10 --max-iters 12` — writes `serving.yaml.engine.gpu_memory_utilization` + `PROFILE_NOTES.md` KV-result block + rehashes profile
7. Thermal pass 1: `uv run python scripts/thermal_replay.py --profile profiles/qwen3.6-27b/v1 --record-floors`
8. Thermal pass 2: `uv run python scripts/thermal_replay.py --profile profiles/qwen3.6-27b/v1 --assert-floors`

**Estimated GPU wall-clock:** ~3.5 h (KV) + 2 h (thermal-1) + 2 h (thermal-2) = ~7.5 h

### Plan 04.1-2: `profiles/gemma-4-31b-it/v1/` bundle

**Scope:** clone-and-retarget from `profiles/gemma-4-26b-a4b-it/v2/` (current Phase-4 MoE bundle). Change:
- `profile.yaml`: `id: gemma-4-31b-it`, `version: v1`, provenance note
- `serving.yaml.engine.model`: `/models/gemma-4-31B-it`
- `serving.yaml.engine.model_hf_id`: `google/gemma-4-31B-it`
- `serving.yaml.engine.served_model_name`: `gemma-4-31b-it`
- `serving.yaml.engine.quantization`: `fp8` (runtime quant of BF16 weights, same pattern as existing 26B v2)
- Container: `vllm/vllm-openai:gemma4-0409-arm64-cu130` (same as Phase 4 v2, same local digest `sha256:db59febc6c47...`)
- Tool_call_parser: `gemma4` + reasoning_parser `gemma4` (same as v2)
- Keep `quirks.strip_thinking_tags: true` — the Gemma 4 channel-bleed strip applies equally to dense
- Keep `load_format: safetensors` (upstream vllm-openai image doesn't ship fastsafetensors)
- `gpu_memory_utilization`: seed at 0.45 (more conservative than 0.55 — 30.7 B BF16 weights ≈ 61 GB of 128 GB UMA is a bigger chunk than 26B MoE's 25.67 GB; extra headroom for the bisection to climb from)

**Tasks:**
1. Pull weights: `huggingface-cli download google/gemma-4-31B-it --local-dir /models/gemma-4-31B-it`
2. Write profile bundle + hash + validate
3. Smoke test via `start_emmy.sh`
4. KV bisection (expect a noticeably lower final value than 0.86 — dense weights leave less KV headroom)
5. Thermal pass 1 (`--record-floors`)
6. Thermal pass 2 (`--assert-floors`)

**Estimated GPU wall-clock:** ~3.5-5 h (KV; boot is ~7 min/iter × more iters expected because the weights take longer and the bisection range is narrower) + 2 h (thermal-1) + 2 h (thermal-2) = ~7.5-9 h

**Explicit throughput policy (per user directive 2026-04-24):** Gemma-4-31B dense is expected to land in the 6-10 tok/s bandwidth-bound zone. This is **NOT** a failure. Thermal pass = zero preemptions + zero OOM + recorded floors. Throughput observation goes into PROFILE_NOTES.md as informational context for Phase 5 eval, not as a gate.

## Constraints (copied forward from Phase 1 + Phase 4 learnings)

- **Pitfall #1 discipline:** `scripts/find_kv_budget.py` is the ONLY sanctioned writer to `serving.yaml.engine.gpu_memory_utilization`. No hand-edits.
- **Profile immutability (D-02):** everything is a new `v1` dir; no in-place edits to any existing profile.
- **Hash-anchored:** `profile.yaml.hash` populated via `emmy profile hash --write` after every bundle change; validate exits 0 before any smoke test.
- **Air-gap discipline:** both profiles must pass `ci_verify_phase3` (STRICT, zero-outbound) unchanged.
- **Container per family:** Qwen 3.6 → NGC 26.03.post1-py3 (fastsafetensors); Gemma 4 → upstream vllm/vllm-openai:gemma4-0409-arm64-cu130 (plain safetensors, ~7-min cold boot).
- **Probe timeout 900 s** in `scripts/smoke_test.py` (Phase 4 follow-up commit `4c5f5c3`) — already handles both Qwen fastsafetensors AND Gemma plain safetensors cold boots.
- **Throughput NOT a gate.** Zero-preemptions, zero-OOM, thermal-floors-recorded are the gates. Low tok/s is acceptable and expected on dense bandwidth-bound.
- **DEFAULT_VARIANT markers** (Phase 4 follow-up commit `61ef905`): each new profile family dir needs a `DEFAULT_VARIANT` file pointing at v1, or the resolver will fall back to historical preference chain.

## Out of scope

- Changing daily-driver default (deferred to Phase 5 post-eval decision)
- Full Phase 5 eval matrix implementation (only the *stub* in 04.1-3)

## Success criteria

1. `profiles/qwen3.6-27b/v1/` validates cleanly, KV bisection converged, 2×2 h thermal "All floors pass"
2. `profiles/gemma-4-31b-it/v1/` validates cleanly, KV bisection converged, 2×2 h thermal "All floors pass" (regardless of tok/s figure)
3. `/profile qwen3.6-27b` and `/profile gemma-4-31b-it` both swap end-to-end on live DGX Spark via the pi-emmy `/profile` slash path (the same path Phase 4 validated for the MoE siblings)
4. `emmy/routes.yaml` optionally extends with a `dense` role token (or skips if operator prefers keeping routing MoE-only for now)
5. `eval/MATRIX.md` enumerates the four participant profiles for Phase 5

## Known risks

- **Gemma dense may fail KV bisection or thermal** if weights+KV exceed 128 GB UMA. Acceptable fallback: reduce `max_model_len` from 131072 → 65536 or 32768 and re-bisect. Document as a deferral if even that fails — model genuinely doesn't fit on this hardware.
- **Qwen 27B-FP8 quant format** — verify `config.json` from HF repo matches vLLM's expected FP8 shape before writing profile; publisher FP8 can come in multiple flavors (FP8-E4M3 weights with FP32 scales, FP8-E4M3 weights with per-channel int8 scales, etc.). Schema validator should catch mismatches but prefer to confirm pre-clone.
