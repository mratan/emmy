<domain>
## Phase Boundary

Ship a `SubAgentTool` (Claude Code-naming convention `Agent`) that lets parent sessions dispatch focused, depth-1, observable sub-agent tasks. Personas are profile-owned. OTel parent-child trace propagation is mandatory (Pitfall #18 mitigation — sub-agents are observable, not black boxes). TUI nesting renders sub-agent transcripts as collapsible blocks under the parent's tool call.

**v1 scope** (single-model, depth-1, max 2 concurrent):
- Two spawn patterns:
  - **Pattern A (lean):** child reuses parent's `AgentSessionServices` (`~5 ms` instantiation per H8). Default for utility sub-agents (small fetches, one-off greps).
  - **Pattern B (persona):** child gets per-persona services + cwd pointing at persona dir with own `AGENTS.md` (`~50 ms` per H4). Default for long-running research/code-reviewer sub-agents.
- Built-in personas: `research`, `code_reviewer`, `bash_runner` (shipped for all 4 profiles)
- Concurrency cap: `max_concurrent: 2` (hardware-imposed per vLLM concurrency spike — separate from this phase)
- Long-context serialization: parent input tokens > 40 K → dispatcher serializes child dispatch
- Per-child JSONL via `SessionManager.create(cwd, subagentDir)` (or `SessionManager.inMemory()` if `persistTranscript: false`)
- Auto-compaction OFF on children (`child.setAutoCompactionEnabled(false)`)
- Clean disposal in `finally`

**What this phase does NOT include** (deliberately, per spike findings):
- Cross-model sub-agents (Qwen parent → Gemma child requires vLLM sleep-mode swap; deferred to a future v2 phase pending sleep-mode-swap latency measurement)
- Parallel sub-agent dispatch beyond `max_concurrent: 2` (vLLM KV-budget hardware ceiling)
- Depth-2+ recursion (Claude Code caps at 1; same here)
- Non-observable sub-agents (Pitfall #18 — every sub-agent step MUST emit OTel span)
- Pi-mono fork (H1–H8 confirmed structurally feasible without)
- Memory tool integration baked in (sub-agents may opt into memory if Phase 04.4 ships first; not required to land 04.5)

</domain>

### Tool name + dispatch interface
- Name: **`Agent`** (Claude Code naming convention; renamed from `Task` in CC v2.1.63).
- Input schema (TypeBox discriminated union):
  ```ts
  Type.Object({
    subagent_type: Type.Union([Type.Literal("research"), Type.Literal("code_reviewer"), Type.Literal("bash_runner")]),
    description: Type.String({ description: "Short label for the dispatch." }),
    prompt: Type.String({ description: "Task for the sub-agent." }),
    model: Type.Optional(Type.String({ description: "Optional model override (single-model in v1; logs warning if used)." })),
  })
  ```
- Output: **text-only summary** to parent (the validated SOTA pattern; intermediate tool calls do NOT pollute parent context).
- Description (parent-facing, ~30 tokens, pi-minimalism):
  > `Agent` — Dispatch a focused task to a sub-agent. Returns the sub-agent's text summary; intermediate tool calls are not surfaced.

### Two spawn patterns (LOCKED, profile-owned via persona config)

**Pattern A — Lean (shared services):**
```ts
const { session: child } = await createAgentSessionFromServices({
  services: parentServices,                          // SHARED with parent
  sessionManager: SessionManager.inMemory(parentCwd),
  model: parentModel,
  customTools: scopedToolSet,                        // narrowed per persona
  tools: persona.tool_allowlist,                     // pi's allowlist option
});
child.setAutoCompactionEnabled(false);
```
Cost: ~5 ms instantiation, 120 KB RSS (per H8 measurement).

**Pattern B — Persona (per-persona services with parent's cwd; persona AGENTS.md injected via resource-loader override):**
```ts
// Read persona's AGENTS.md from the profile bundle. This is the persona-defining
// content — but we DO NOT use it as the child's cwd (that would point tools at the
// profile bundle and break grep/read on the user's project).
const personaAgentsPath = path.resolve(profilePath, "subagents", persona.persona_slug, "AGENTS.md");
const personaAgentsContent = readFileSync(personaAgentsPath, "utf8");

const subagentServices = await createAgentSessionServices({
  cwd: parentCwd,                                    // child operates on user's project
  authStorage: parentAuthStorage,                    // SHARED (auth is profile-level)
  resourceLoaderOptions: {
    // Inject persona AGENTS.md as an additional project-context file.
    // Pi reads parent's CLAUDE.md/AGENTS.md from cwd PLUS this injected content.
    agentsFilesOverride: (base) => ({
      agentsFiles: [
        ...base.agentsFiles,
        { path: personaAgentsPath, content: personaAgentsContent },
      ],
    }),
  },
});
const { session: child } = await createAgentSessionFromServices({
  services: subagentServices,
  sessionManager: SessionManager.create(parentCwd, persona.persistDir ?? undefined),
  model: personaModel,
  customTools: scopedToolSet,
  tools: persona.tool_allowlist,
});
child.setAutoCompactionEnabled(false);
```
Cost: ~50 ms instantiation (per H4 measurement; H4 used cwd-divergence to prove per-session prompts differ — same divergence is achieved here via `agentsFilesOverride` without sacrificing tool cwd).

**Why `cwd: parentCwd` and NOT `parentCwd/persona_dir`:** pi-mono's `cwd` is used both for project-context discovery (AGENTS.md/CLAUDE.md) AND for tool default working directory (bash, read relative paths). If we pointed `cwd` at the persona bundle, the child's `read`/`grep`/`find`/`ls` would operate on `profiles/<name>/<v>/subagents/<persona>/` — a directory containing only AGENTS.md. That defeats the research persona's whole purpose ("find usages of customTools in the codebase"). The fix is to decouple persona-prompt-source from tool-cwd: use `parentCwd` for tools, inject persona AGENTS.md via `resourceLoaderOptions.agentsFilesOverride` (or equivalently `appendSystemPrompt: [personaAgentsContent]` if simpler).

Both patterns: child must be `dispose()`'d in `finally`.

### Persona schema (`harness.yaml subagents.personas.<name>`, NEW)

### Persona dir layout (Pattern B only)
```
profiles/<name>/v<N>/subagents/<persona>/
  AGENTS.md                  # persona's system prompt (~100-300 tokens, pi-minimalism)
  prompts/                   # optional: persona-specific prompt fragments
```
**Each `persona_dir` ships with all 4 shipped profiles.** That's 3 personas × 4 profiles = 12 markdown files.

### OTel parent-child propagation (Pitfall #18 contract)
Every sub-agent step MUST emit OTel span parented to the `Agent` tool's span:
```ts
const tracer = trace.getTracer("emmy-subagent");
return await tracer.startActiveSpan(
  `subagent.${persona.name}`,
  { attributes: { "gen_ai.agent.name": persona.name, "gen_ai.agent.id": persona.id } },
  async (span) => {
    try {
      const parentCtx = context.active();
      return await context.with(parentCtx, async () => {
        const { session: child } = /* spawn per Pattern A or B */;
        try {
          return await runOneTurnReturningText(child, params.prompt);
        } finally {
          child.dispose();
        }
      });
    } finally {
      span.end();
    }
  },
);
```
- H5 confirmed: `AsyncLocalStorage` survives pi's tool dispatch → `context.with(parentCtx, ...)` works without explicit injection
- Trace tree: `parent_session → Agent_tool_span → subagent.<name> span → child_invoke_agent → child_chat_completion`
- GenAI semconv attributes: `gen_ai.agent.{id, name, version}`, `gen_ai.conversation.id`
- Per-request `emmy.subagent.parent_span_id` (span tree handles the relationship; this is informational redundancy for debugging)

### Concurrency governor
- Default `max_concurrent: 2` (hardware-imposed per vLLM concurrency spike)
- Hard rule: parent input tokens > 40 K → sub-agents serialize (no parallel dispatch)
- Parent suspend during child run is OPTIONAL in v1 (defer to v2 if measurement shows benefit)
- Telemetry event `agent.dispatch.serialized` when long-context rule fires
- Telemetry event `agent.dispatch.queued` when concurrent cap hit

### TUI nesting (custom Ink component — only piece pi doesn't help with)
- Pi has no off-the-shelf collapsible-nested-turn widget
- Build custom Ink component that subscribes to child session events and re-emits `invalidate()` on each child turn
- Render: parent's `Agent` tool call shows a header (`▶ Agent: research — "find usages of customTools"`) with collapsible child-turn block beneath; child's tool calls visible in the block but indented + dimmed
- Default state: collapsed; user can expand
- Component lives in `packages/emmy-ux/src/components/subagent-block.tsx` (or .ts depending on pi-tui conventions)

### Per-child JSONL location (LOCKED)
- Pattern A (lean) default: `SessionManager.inMemory(parentCwd)` (no disk write)
- Pattern B (persona) default: `SessionManager.create(parentCwd, parentDir/.subagents/<persona>-<n>)`
- Persona's `persist_transcript: true` overrides Pattern A to also use `SessionManager.create`
- Per-child JSONL filename includes parent's `trace_id` for forensics linkage
- Sidecar JSONL `session-<id>.subagents.jsonl` in parent's session dir lists all child trace_ids spawned during that session (linked by `parent_span_id`)

### Tool allowlist below pi's defaults
H2 confirmed: customTools layer per-session with no leakage. Pi's `tools` allowlist (in `CreateAgentSessionFromServicesOptions`) narrows below pi's default `[read, bash, edit, write]`. v1 personas exercise this:
- `research`: `[read, grep, find, ls]` (no bash, no edit, no write)
- `code_reviewer`: `[read, grep]`
- `bash_runner`: `[bash, read]`

V3 verification confirms this is enforced (model trying to call disallowed tool gets rejection visible in child transcript).

### Disposal contract
- Every Pattern A and Pattern B spawn site MUST `dispose()` in finally
- H8 confirmed: 50 spawn/dispose cycles, 0 handle leak, +5.9 MB RSS, 2.4 ms/cycle — pi-mono is well-behaved
- No additional cleanup workaround needed

### Pitfall #22 awareness (architecture-aware verification)
H9 spike (2026-04-26) discovered vLLM prefix caching is silently 0% hit rate on Mamba-hybrid Qwen 3.6 35B-A3B. The integration sketch's original "Pattern A = free prefix-cache hits" argument has been **withdrawn**. The instantiation-cost argument (5 ms vs 50 ms) still stands. V8 (real-deal E2E) measurement is informational on prefix-cache hit rate; it is NOT a hard pass criterion (per Pitfall #22).

### Claude's Discretion
- Internal file layout *within* `packages/emmy-tools/src/subagent/` — proposed module names (`index.ts`, `dispatcher.ts`, `persona-loader.ts`, etc.) may be consolidated/split per planner's judgment.
- Exact text of persona `AGENTS.md` files — planner may draft tight 100-300 token prompts per persona; pi-minimalism applies.
- Whether to test V5 (concurrent children no cross-talk) with 2 or 3 concurrent children — 2 is the cap, 3 is over-cap and exercises rejection. Both worth testing; planner picks scope.
- TUI nesting collapsed-by-default vs expanded-by-default — pick the more useful default; document the trade-off.
- The `SessionStartEvent` metadata for sub-agents (per `AgentSessionConfig.sessionStartEvent`) — planner decides what's useful to capture.

## Specific Targets

### Success criteria (V1–V8 LOCKED, mirrors INTEGRATION-SKETCH.md §4)

- **V1 — Pattern A spawn+execute:** lean child runs a task, returns text, disposes; <100 ms wall under faux model
- **V2 — Pattern B spawn+execute:** persona child has different `systemPrompt` from parent; runs a task; disposes
- **V3 — Tool allowlist enforced:** persona configured for `[read, grep]` cannot call `bash`; rejection visible in child's transcript
- **V4 — OTel parent-child propagation:** parent span A → SubAgentTool span B → child invoke_agent span C all share `trace_id`. Run with @emmy/telemetry's real exporter
- **V5 — Concurrent children:** parent dispatches 2 children in parallel; both complete; no cross-talk
- **V6 — Long-context serialization rule:** parent input tokens > 40 000 → dispatcher serializes child dispatch; `agent.dispatch.serialized` telemetry event fires
- **V7 — Auto-compaction-off on children:** child built with `setAutoCompactionEnabled(false)` does NOT compact mid-run even at high context
- **V8 — Real-deal E2E:** live Qwen 3.6 35B-A3B v3.1; parent fires `Agent({subagent_type: "research", prompt: "find usages of customTools"})` → child does grep+read → returns summary → parent ingests. Verify OTel trace tree end-to-end. Prefix-cache hit rate is informational only (Pitfall #22)

**V1–V7 are blocking. V8 is the smoke-test forcing-function.**

## Deferred Ideas

These were discussed in pre-phase and explicitly deferred:

- **Cross-model sub-agents (Qwen parent → Gemma child)** — requires vLLM sleep-mode swap, not parallel co-resident engines (memory math: 110% of UMA at gmu=0.55 each). Future v2 phase pending sleep-mode swap latency measurement.
- **Parallel sub-agent dispatch beyond max_concurrent: 2** — vLLM KV-budget hardware ceiling on single GPU. Concurrency spike's recommendation: long-context parents serialize anyway.
- **Depth-2+ nesting** — Claude Code caps at 1; same here. Sub-agents cannot spawn sub-agents.
- **Real-provider full E2E from spike** — H9 hit the wire directly via `fetch()` rather than through pi-mono's real-provider chain. The structural plumbing is validated by H1–H8; the real-provider integration is Phase 04.5 implementation work.
- **Memory tool integration baked in** — sub-agents may opt into the memory tool if Phase 04.4 ships first; not a hard dependency for this phase.
- **`tools` allowlist for the parent itself** — pi documents the option; not exercised in v1 (parent gets pi defaults + customTools).
- **Sub-agent's own auto-compaction with custom prompt** — out of scope; children are short-lived and don't compact.
- **Per-persona model override beyond profile** — config exists but not wired in v1 (single-model only).
