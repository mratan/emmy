# Phase 04.5 Plan 04: Concurrency governor + V5/V6/V7 BEHAVIORAL Summary

`ConcurrencyGovernor` caps Agent dispatches at maxConcurrent (LOCKED default 2),
serializes when parent input tokens exceed 40K (LOCKED), and rejects 3rd
concurrent dispatch with `agent.dispatch.rejected` (I3). All three telemetry
events fire on the right paths. Dispatcher wires governor.acquire() in front of
withSubagentSpan with OTel context capture+re-attach to survive Bun's AsyncHooks
loss-across-await (a debug-discovered behavior fixed inline).

## What was built

### Source — governor.ts

- `ConcurrencyGovernor.acquire({ parentInputTokens? })` returns `{release, serialized, waited_ms}`.
- `effectiveCap`: long-context branch returns `{cap: 1, serialized: true}` when tokens > threshold.
- LOCKED defaults: maxConcurrent=2, longContextSerializeThresholdTokens=40000, rejectOverCap=true (I3 LOCKED).
- Telemetry events all match CONTEXT.md §decisions LOCKED shapes:
  - `agent.dispatch.serialized` — fired when long-context branch triggers; payload includes `parent_input_tokens` + `threshold`.
  - `agent.dispatch.rejected` — fired (I3) when over-cap; payload includes `reason: "over_cap"`, `in_flight`, `cap`. Throws `Error` mentioning "concurrent dispatch cap".
  - `agent.dispatch.queued` — fired when an acquire waited > 1ms (queue mode); payload includes `wait_ms`, `in_flight`, `cap`.
- Release is idempotent (second call no-op).

### Source — dispatcher.ts wiring

- `resolveGovernor(opts)`: returns `opts.governor` or lazily creates a default per-`opts` instance via WeakMap (one governor per parent session/tool).
- `dispatchSubAgent` body (NEW order):
  1. Capture `otelContext.active()` BEFORE governor.acquire's await (preserves parent OTel context across Bun AsyncHooks loss).
  2. `governor.acquire({ parentInputTokens: opts.parentInputTokens?.() })` — may reject (I3) and propagates upward.
  3. Inside try: `otelContext.with(parentCtx, () => withSubagentSpan(...))` so the subagent span is correctly parented to `agent.tool.Agent` regardless of ALS loss.
  4. `release()` in finally.
- The `dispatchSubAgentInner` helper (services + spawn + child dispose) unchanged.

### Source — types.ts additions

- `parentInputTokens?: () => number` — getter evaluated per-dispatch (parent context grows over a session).
- `governor?: ConcurrencyGovernor` — optional injection; default lazily created.

## Deviations from Plan

**1. [Rule 1 — bug] Bun AsyncHooks loses OTel ALS state across some awaits.**
   - Found during: V4 E2E test from Plan 04.5-03 broke after governor.acquire was inserted before withSubagentSpan. The subagent span's traceId diverged from the parent's because `tracer.startActiveSpan` lost the active context across the governor's await boundary.
   - Diagnosis: empirically verified with a standalone debug script — `await new Promise(r => setTimeout(r, 10))` is enough for `trace.getActiveSpan()` to return undefined inside an active span callback.
   - Fix: capture `otelContext.active()` BEFORE governor.acquire's await, then `otelContext.with(parentCtx, () => withSubagentSpan(...))` to re-attach inside the subagent span block. This works because OTel's API-level context.with explicitly applies the snapshot regardless of ALS state.
   - Verification: V4 E2E test from Plan 04.5-03 + V6 long-context tests all pass.

**2. [Rule 3 — additional functionality] V7 BEHAVIORAL via session-factory test seam.**
   - Found during: Task 3 first attempt at driving compaction via faux contextWindow=8192 + 20 turns.
   - Issue: faux providers don't fully participate in pi's context-pressure calculations; pi never decided to compact even with auto-compaction ENABLED in the counter-baseline. The plan's safety-bound fallback `child.shouldCompact?.()` doesn't exist as a public method either.
   - Fix: pivoted to a stronger, more direct verification using the existing `__setSessionFactoryForTests` seam (Plan 04.5-01):
     - Test 1 spies on `setAutoCompactionEnabled` calls — proves the dispatcher calls it with `false` exactly once per dispatch.
     - Test 1 also subscribes to ALL child events and asserts zero `compaction_start`/`compaction_end` events fire — the actual contract Pitfall #18 cares about.
     - Test 2 runs 20 sequential dispatches and asserts both invariants hold across all of them.
   - This is stronger than the plan's compaction-pressure approach because it works regardless of whether faux providers can trigger pi's compaction-eligibility heuristic. If a future SDK change makes setAutoCompactionEnabled a no-op OR starts firing compaction events anyway, both tests catch it.

**Total deviations:** 2 (1 Rule 1 OTel context preservation, 1 Rule 3 V7 verification approach).
