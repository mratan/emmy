## Ambiguous-match error string format

Used by str_replace when `old_str` matches more than once in the file. Plan 04.4-04 air-gap stress run can pattern-match this:

```
Error (memory.ambiguous_match): old_str matched <N> times at lines <a, b, c>. Resubmit with more context to disambiguate.
```

Where `<N>` is the count and `<a, b, c>` is the comma-separated 1-indexed line-number list. The structured error also carries `details: { matchCount, lineNumbers }`.

## Deviations from Plan

**`rmSync` → `rmdirSync` + `unlinkSync` (technical fix).** Plan body uses `rmSync(absPath, { recursive: false })` for both files and empty dirs. On bun 1.3.13 this throws EFAULT on the empty-dir path (likely a libuv quirk on this kernel). Switched to the unambiguous `rmdirSync` for empty dirs and `unlinkSync` for files — same external behavior, no syscall ambiguity.

**`MemoryToolOpts.cwd` propagation.** Plan body uses `opts.cwd ?? process.cwd()` once at factory construction. Done as written. The dispatcher recomputes scopeRootAbs per-call from this captured value, which is what tests expect.

**Removed obsolete plan-01 smoke test.** Plan 01's `quotas.test.ts` had a smoke test asserting the dispatcher returns `memory.not_implemented`. Plan 02's whole point is filling that body, so the smoke test now asserts a successor behavior (`memory.not_found` for non-existent file) — same level of confidence in the dispatcher being live.

**Total deviations:** 0 architectural; 3 minor (Bun-syscall fix + obsolete-test refresh + carrying-forward `cwd` semantics).
