    Six gates that require human-driven measurement; cannot run inside the executor wave.
    Source of truth: MEMORY-TOOL-SPEC.md §10 + COMPACTION-DESIGN.md §8.

    ## V1 — Memory adoption ≥ 60% (Qwen 35B-A3B v3.1)

    **Fixture:** 20-task batch from `.planning/eval/memory-adoption-fixture.md` (or compose if absent).
    Tasks chosen so each presents a "would benefit from a note" moment — e.g.:
    - "Find and fix the off-by-one bug in `src/parsers/tokenizer.ts`"
    - "Add a new endpoint following the existing CRUD convention"
    - "What was that hashline edit pattern again?"
    Each task is a fresh session; memory tool enabled; project + global scope clean before task 1, then accumulating.

    **Protocol:**
    1. `pi-emmy --profile qwen3.6-35b-a3b/v3.1` (no `--no-memory`)
    2. Run all 20 tasks; record each session's transcript via the JSONL telemetry
    3. For each task, hand-rate:
       - Did the model `view /memories/...` before the work? (Y/N)
       - Did it write a note via `create` or `str_replace` when a discovery was made? (Y/N — only when applicable)
    4. Adoption = (sessions with at least one `view` call) / 20
    5. **Pass:** adoption ≥ 60%
    6. **Fail behavior:** revise tool description in plan 04.4-01 (see CONTEXT.md "Calibration protocol: measurement-driven expansion only"); re-run with v2 description.

    **Resume signal:** `v1 memory adoption green: <observed_pct>%` OR `v1 memory adoption FAIL — observed <pct>%, recommend description revision`

    ## V2 — Memory write discipline ≥ 70% load-bearing

    **Fixture:** every `create` and `str_replace` op in the V1 run's JSONL telemetry.
    Reuses the V1 sessions; no new run needed.

    **Protocol:**
    1. `cat ~/.emmy/telemetry/*.jsonl | jq 'select(.event=="memory.op" and (.["emmy.memory.command"]=="create" or .["emmy.memory.command"]=="str_replace"))'` → bucket of write ops
    2. For each write: hand-rate "load-bearing" — would a future session NEED this note to do its job? (Y/N)
       - YES: API quirk, project convention, hard-won discovery
       - NO: narrative ("then I fixed the bug"), already-obvious, transient state
    3. Discipline = (load-bearing writes) / (total writes)
    4. **Pass:** discipline ≥ 70%
    5. **Fail behavior:** the v1 description didn't sufficiently discourage narrative; expand description with "write notes only when a discovery would help a future session" sterner phrasing. See CONTEXT.md calibration protocol.

    **Resume signal:** `v2 memory discipline green: <observed_pct>%` OR `v2 memory discipline FAIL — observed <pct>%, narrative writes: <samples>`

    ## V3 — Memory rot protection 100% — CRITICAL DIMENSION

    **Fixture:** 5 planted contradicting notes pre-loaded into `/memories/project/`:
    1. `notes/api-format.md`: "Always use `requestId` in JSON payloads (camelCase)" — but the code actually uses `request_id` (snake_case)
    2. `notes/auth.md`: "Tokens are signed with HS256" — but `auth.ts` actually uses RS256
    3. `notes/db-pool.md`: "Database max connections is 50" — actual config says 200
    4. `notes/env-var.md`: "Set `LOG_LEVEL=verbose` to enable debug" — actual code reads `DEBUG=1`
    5. `notes/route.md`: "POST /api/v1/users for create" — actual route is `POST /users` (no /api/v1)

    Pre-load these into a project's `.emmy/notes/` then ask the model 5 questions, one per planted contradiction (e.g. "What's the JSON field convention here? Should I use camelCase or snake_case?").

    **Protocol:**
    1. Set up the planted notes in a fresh test project under `/tmp/v3-rot-test/`
    2. `cd /tmp/v3-rot-test && pi-emmy --profile qwen3.6-35b-a3b/v3.1`
    3. Ask each of the 5 questions in a fresh session (or one session — both are valid V3 runs)
    4. For each, observe: did the model TRUST the note blindly, or did it CHECK the code (read source file) before answering?
    5. **Pass:** 5/5 — model checked code AND surfaced the contradiction OR followed the code-truth answer
    6. **Fail behavior:** rot protection is the CRITICAL DIMENSION (CONTEXT.md). If <5/5, the tool description needs sterner "verify before trusting" language; consider adding `last_updated:` line to note metadata (deferred per CONTEXT.md Claude's discretion).

    **Resume signal:** `v3 memory rot green: 5/5` OR `v3 memory rot FAIL — <N>/5, samples where model trusted note: <list>`
