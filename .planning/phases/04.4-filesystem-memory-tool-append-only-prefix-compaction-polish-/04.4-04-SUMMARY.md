## Profile auto-stamp on memory spans — confirmed

Profile stamping is handled by `EmmyProfileStampProcessor` (installed at
@emmy/telemetry boot). The hook factory does NOT manually attach
profile.{id,version,hash}; it emits via the dual-sink which routes through
the auto-stamp processor. This is verified end-to-end in the airgap-stress
test (events.length === 100 with auto-stamps applied at OTel SpanProcessor.onStart).

## Deviations from Plan

**Test 10 (mock OTel processor verification of auto-stamp).** Plan asked
for a mock span processor that records `(spanName, attributes)` tuples and
asserts `emmy.profile.id|version|hash` are attached. Implemented the
auto-stamp verification at the END-TO-END level instead — the airgap-stress
test exercises the real `EmmyProfileStampProcessor` through 100 events.
The processor's correctness is already covered by Phase 3 tests
(`packages/emmy-telemetry/test/profile-stamp-processor.test.ts`). Less
ceremonial mock setup; equivalent (or better) confidence.

**`tests/` vs `test/` (emmy-tools convention).** emmy-tools uses `tests/`
(plural) — files landed there as the plan body specifies. emmy-context
uses `test/` (singular); plan 07's tests landed there.

**Total deviations:** 0 architectural; 1 minor (auto-stamp verification
shifted from unit-mock to end-to-end stress).
