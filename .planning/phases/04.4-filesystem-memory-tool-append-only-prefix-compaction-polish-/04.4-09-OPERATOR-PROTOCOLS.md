## V1 — Memory adoption ≥ 60% (Qwen 35B-A3B v3.1)

**Fixture:** 20-task batch — pick tasks that present a "would benefit from a note"
moment. Examples:
- "Find and fix the off-by-one bug in `src/parsers/tokenizer.ts`"
- "Add a new endpoint following the existing CRUD convention"
- "What was that hashline edit pattern again?"
- "We had to use TextEncoder for prefix-hash bytes — explain why"

Each task is a fresh session; memory tool enabled (default); project + global scope
clean before task 1, then accumulating.

**Protocol:**
1. `pi-emmy --profile profiles/qwen3.6-35b-a3b/v3.1` (no `--no-memory`)
2. Run all 20 tasks; record each session's transcript via JSONL telemetry
3. For each task, hand-rate:
   - Did the model `view /memories/...` before the work? (Y/N)
   - Did it write a note via `create` or `str_replace` when a discovery was made? (Y/N — only when applicable)
4. Adoption = (sessions with at least one `view` call) / 20
5. **Pass:** adoption ≥ 60%
6. **Fail behavior:** revise tool description in plan 04.4-01 per CONTEXT.md
   "Calibration protocol: measurement-driven expansion only"; re-run.

**Resume signal:** `v1 memory adoption green: <observed_pct>%` OR
`v1 memory adoption FAIL — observed <pct>%, recommend description revision`

## V2 (memory) — Memory write discipline ≥ 70% load-bearing

**Fixture:** every `create` and `str_replace` op in the V1 run's JSONL telemetry.
Reuses the V1 sessions; no new run needed.

**Protocol:**
```bash
cat ~/.emmy/telemetry/*.jsonl | \
  jq 'select(.event=="memory.op" and (.["emmy.memory.command"]=="create" or .["emmy.memory.command"]=="str_replace"))' \
  > /tmp/v2-write-ops.json
```

For each write: hand-rate **load-bearing** vs **narrative**:
- **YES (load-bearing):** API quirk, project convention, hard-won discovery,
  counter-intuitive finding, dead-end others should skip
- **NO (narrative):** "then I fixed the bug", already-obvious facts, transient state

Discipline = (load-bearing writes) / (total writes)

**Pass:** discipline ≥ 70%
**Fail behavior:** description didn't sufficiently discourage narrative; expand
description with sterner phrasing per CONTEXT.md calibration protocol.

**Resume signal:** `v2 memory discipline green: <observed_pct>%` OR
`v2 memory discipline FAIL — observed <pct>%, narrative writes: <samples>`

## V3 — Memory rot protection 100% — CRITICAL DIMENSION

**Fixture:** 5 planted contradicting notes pre-loaded into a fresh test project's
`/memories/project/`. The 5 plants are designed so each is checkable against a
specific code file in the test fixture project:

1. `notes/api-format.md`: "Always use `requestId` in JSON payloads (camelCase)" —
   but the code actually uses `request_id` (snake_case)
2. `notes/auth.md`: "Tokens are signed with HS256" — but `auth.ts` actually uses RS256
3. `notes/db-pool.md`: "Database max connections is 50" — actual config says 200
4. `notes/env-var.md`: "Set `LOG_LEVEL=verbose` to enable debug" — actual code reads `DEBUG=1`
5. `notes/route.md`: "POST /api/v1/users for create" — actual route is `POST /users`
   (no /api/v1)

**Protocol:**
1. Set up the planted notes in a fresh test project under `/tmp/v3-rot-test/`
   with corresponding source files containing the truth
2. `cd /tmp/v3-rot-test && pi-emmy --profile profiles/qwen3.6-35b-a3b/v3.1`
3. Ask each of the 5 questions in a fresh session (or one session — both valid):
   - "What's the JSON field convention here? Should I use camelCase or snake_case?"
   - "What signing algorithm does `auth.ts` use?"
   - "What's the max database connections this app uses?"
   - "How do I enable debug logging?"
   - "What's the user-creation endpoint path?"
4. For each, observe: did the model TRUST the note blindly, or did it CHECK the
   code (read source file) before answering?
5. **Pass:** 5/5 — model checked code AND surfaced the contradiction OR followed
   the code-truth answer
6. **Fail behavior:** rot protection is the CRITICAL DIMENSION. If <5/5, the tool
   description needs sterner "verify before trusting" language; consider adding
   `last_updated:` line to note metadata (deferred per CONTEXT.md Claude's discretion).

**Resume signal:** `v3 memory rot green: 5/5` OR
`v3 memory rot FAIL — <N>/5, samples where model trusted note: <list>`

## V8 (memory) — 1-hour real-deal live session

**Fixture:** any non-trivial coding task that takes ~1 hour with the agent.
Examples:
- "Add an OAuth2 PKCE flow to the sample app + tests"
- "Refactor the parser module to support streaming"
- "Build a Phase 5 dashboard component end-to-end"

Operator chooses one.

**Protocol:**
1. `pi-emmy --profile profiles/qwen3.6-35b-a3b/v3.1` (default profile, fresh session)
2. Drive the agent through the task for ~60 minutes
3. Observe organically:
   - 1–5 memory notes created across the session (not 0, not 20)
   - No narrative writes ("then I added the import")
   - No quota errors in JSONL: `grep memory.quota_exceeded ~/.emmy/telemetry/*.jsonl`
   - No rot incidents (model didn't trust a stale note over fresh code)
   - All write ops have `emmy.memory.scope` and load-bearing reasoning
4. **Pass:** all 5 sub-criteria green
5. **Fail behavior:** record which sub-criterion failed; specific revision path
   depends on the failure (typically loops back to V1/V2/V3 fixes)

Quick post-session checks:
```bash
grep -E 'memory.quota_exceeded' ~/.emmy/telemetry/*.jsonl   # → 0 matches
cat ~/.emmy/telemetry/*.jsonl | \
  jq 'select(.event=="memory.op" and .["emmy.memory.command"]=="create")' | \
  jq -s 'length'                                            # → 1..5
```

**Resume signal:** `v8 memory real-deal green` (counts: <N notes>; quota errors: 0;
rot incidents: 0) OR FAIL markers per failed sub-criterion

## V2 (compaction) — Compaction summary quality (3-axis hand-rate)

**Fixture:** 10 multi-turn coding sessions with organic compaction. These can be
the V8 session + 9 others (ideally: 3 each across qwen3.6-35b-a3b/v3.1,
qwen3.6-27b/v1.1, and gemma-4-26b-a4b-it/v2; the dense Gemma 31B is single-digit
tok/s and may not produce enough turns to organically compact within a sitting —
exclude it from V2 unless an extended run is feasible).

**Protocol:**
1. Run 10 multi-turn sessions, each long enough that `shouldCompact()` triggers
   at least once (≥ 75% of `max_input_tokens` = 114688 → ~86K tokens)
2. For each compaction, fetch the summary:
```bash
cat ~/.emmy/telemetry/*.jsonl | \
  jq 'select(.event=="session.compaction.complete") | {ts, profile, summary_tokens, summary}' \
  > /tmp/v2-compaction-summaries.json
```
3. Hand-rate each summary on three axes:
   - **Goal preservation** (≥ 80% target): does the summary preserve every
     explicit goal raised in the elided turns?
   - **Decision preservation** (≥ 70%): does it preserve decisions made?
     (e.g., "we chose library X over Y because Z")
   - **Error preservation** (100% — CRITICAL): does it preserve every error
     encountered + how it was resolved?
4. Average across 10 sessions per axis

**Pass:** goal ≥ 80%, decision ≥ 70%, error 100%
**Fail behavior:** which axis missed determines the prompt revision per
CONTEXT.md calibration plan:
- Goal miss → add "explicit goals" emphasis
- Decision miss → add "decisions made and their rationale"
- Error miss → add "errors and their resolutions" stronger phrasing

The current v1 prompt ALREADY names all three; if any axis misses despite that,
the prompt may need sectioning (deferred per CONTEXT.md "Compaction prompt
sectioning" — Pitfall #5 trap).

**Resume signal:** `v2 compaction quality green: goal=<pct>% decision=<pct>%
error=<pct>%` OR FAIL markers per axis

## V8 (compaction) — 2-hour real-deal session + arch-aware prefix-cache hit rate

**Fixture:** sustained coding work on Qwen 3.6 35B-A3B v3.1 for ~2 hours.

**Protocol:**
1. `pi-emmy --profile profiles/qwen3.6-35b-a3b/v3.1` (fresh session)
2. Drive sustained work for ~2 hours; let compaction trigger organically
3. Continuously check vLLM `/metrics` for preemption / OOM:
```bash
curl -s http://127.0.0.1:8002/metrics | grep -E "preempt|oom"
# → preempted_total{...} 0, out_of_memory_total{...} 0 throughout
```
4. After at least one compaction has fired, read prefix-cache hit rate:
```bash
curl -s http://127.0.0.1:8002/metrics | grep prefix_cache
# Compute: hits / (hits + misses) post-compaction
```
5. **Architecture-aware gate (per Pitfall #22):**
   - Qwen 35B-A3B is **Mamba-hybrid** → "measure-and-acknowledge"; record the
     hit rate, do NOT fail-gate (vLLM marks Mamba prefix caching experimental
     and silently no-ops)
   - For attention-only profiles (qwen3.6-27b dense, gemma-4-31b-it dense),
     the same protocol with Qwen 27B as substrate → **hard ≥ 80%** post-compaction
6. **Pass criteria for Mamba-hybrid path:**
   - 0 preemptions across the 2-hour run
   - 0 OOM kills
   - At least one compaction fired and produced a non-empty summary
   - Prefix-cache hit rate recorded (any value — informational only per Pitfall #22)
7. **Pass criteria for attention-only path (optional V8' on Qwen 27B):**
   - All Mamba-hybrid criteria above
   - Plus prefix-cache hit rate ≥ 80% post-compaction

**Resume signal (Mamba-hybrid):** `v8 compaction real-deal green: 0 preempt/oom,
hit_rate=<pct>% (mamba-hybrid measure-and-acknowledge)` OR FAIL with cause

**Resume signal (attention-only optional):** `v8 compaction real-deal green
(attention-only): hit_rate=<pct>% ≥ 0.80`

  executor wave. The wave-4 plan (04.4-09) lands this runbook + a STATE.md
  closeout-pending stamp; the operator runs the protocols and types resume
  signals; the planner re-engages to flip the ledger and close the phase.

  hand-rating ~30 min, V8 memory + V2 compaction can ride the same session,
  V8 compaction = 2 hr standalone).

  memory + V2 compaction together (single 1-hour session), V8 compaction last
