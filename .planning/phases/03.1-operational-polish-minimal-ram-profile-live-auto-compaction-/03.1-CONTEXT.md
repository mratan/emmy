## Phase Boundary

Phase 3.1 ships **three surgical fixes + one new capability + daily-ops docs**:

**Fix A — RAM headroom (D-29):** `gpu_memory_utilization=0.88` on DGX Spark UMA (128 GB unified pool) reserves ~112 GB for GPU/KV, which steals from system RAM and produces 116/125 GB used + 8 GB swap during idle emmy-serve. CLAUDE.md Pitfall #3 explicitly warned ("Start at 0.75") but the Phase 1 KV-finder landed at 0.88 because the bisection objective was "highest value that doesn't preempt" rather than "highest value that leaves OS headroom". Drop to ~0.55 while KEEPING `max_model_len=131072` (user constraint: don't shrink context). Expected outcome: emmy-serve footprint ~50-70 GB, >40 GB system headroom, no swap pressure.

**Fix B — Live compaction + manual escape (D-30, D-31, D-32):** Plan 03-03 shipped `emmyCompactionTrigger` on `pi.on("turn_start")` with D-14 preservation + D-16 fallback + D-12 fail-loud. SC-2 green in stub mode. But the trigger's `engine.summarize()` injection point was never wired to pi's native `ctx.compact(options)` method (verified at `dist/core/extensions/types.d.ts:224`). Result: agent hit the 114,688-token hard wall during a real research session; compaction never fired; no manual escape (`/clear` attempted as a pi slash command failed — `/clear` is not a pi 0.68 built-in, and at 114,689 tokens the session couldn't process any further input). Plan 03.1-01 wires the turn_start handler to call `ctx.compact({customInstructions})` directly and adds two pi slash commands: `/compact [optional instructions]` and `/clear`.

**Fix C — Local research egress (D-33, D-34, D-35, D-36):** v3 profile ships a 5-host doc allowlist (docs.python.org, developer.mozilla.org, docs.vllm.ai, huggingface.co, docs.langfuse.com). Too narrow for a daily-driver research loop. The thesis is "NO CLOUD INFERENCE" — the LLM always runs locally on emmy-serve. "NO OUTBOUND TRAFFIC AT ALL" was a broader interpretation that conflates the two. Phase 3.1 refines the thesis: local LLM is non-negotiable; local-first EGRESS (a single self-hosted SearxNG sibling to Langfuse) is acceptable for web research. SearxNG is self-hosted at `127.0.0.1:8888`, auditable, and the harness only talks to SearxNG (which in turn talks to Google + DDG + Brave + Bing upstreams). New `web_search` tool hits the loopback SearxNG. `web_fetch` gets returned-URL bypass semantics: any URL returned by `web_search` in the last N seconds/results is fetchable without allowlist entry. Badge semantics split into three states: `OFFLINE OK` (green) when SearxNG is down/disabled; `LOCAL LLM · WEB` (yellow) when SearxNG is up and web_search is live; `CLOUD INFERENCE` (red) reserved for actual thesis violation (non-loopback inference detected — never fires today).

**New artifact — Profile v3.1 (D-37):** Sibling of v3 per D-02 immutability. v3 stays byte-identical. v3.1 = v3 + (a) tuned `serving.yaml` with `gpu_memory_utilization: 0.55`, `max_num_batched_tokens: 8192`; (b) updated `harness.yaml` with `tools.web_search` block + `tools.web_fetch` bypass policy (allowlist retained for backwards compat); (c) `PROFILE_NOTES.md` provenance for D-29..D-36; (d) new prompts nothing — compaction prompts unchanged from v3.

**New docs (D-38 sequencing, Plan 03.1-03):** Top-level `README.md` for the project (prerequisites + quickstart + stack boot/teardown + profile switching + troubleshooting); `docs/runbook.md` for daily ops; `CLAUDE.md` revised to split "No cloud INFERENCE" (hard invariant, air-gap tested) from "Local-first EGRESS" (SearxNG is the only outbound component, auditable, loopback-only from harness's POV).

### D-29 — RAM target
Drop `serving.yaml.engine.gpu_memory_utilization` from 0.88 → 0.55 in v3.1. KEEP `max_model_len=131072` (user constraint: don't reduce model capability). Drop `max_num_batched_tokens` from 16384 → 8192 (still well-above the "interactive latency" threshold from STACK.md; reduces CUDA workspace footprint). Target emmy-serve footprint ~50-70 GB, leaving >40 GB system headroom. Post-deploy validation: `free -h` MUST show `available ≥ 40G` after 10 min of warm emmy-serve + Langfuse compose. v3 profile is NOT modified.

### D-30 — Live compaction actually fires
`emmyCompactionTrigger` at `packages/emmy-context/src/compaction.ts` already contains D-11 soft-threshold check, D-14 preservation pre-filter, D-12 hard-ceiling, D-16 fallback. The `pi-emmy-extension.ts` `pi.on("turn_start", ...)` handler invokes the trigger but today's `engine.summarize()` default throws ("not configured"). D-30 locks the live wiring: the turn_start handler MUST call `ctx.compact({customInstructions})` directly (pi 0.68 ExtensionContext.compact method, verified at `dist/core/extensions/types.d.ts:224`). Emmy retains its preservation classifier (runs BEFORE the ctx.compact call to annotate which entries are preservation-critical), but the actual summarization round-trip flows through pi's native compaction engine — which already reuses the active Model from the session (the same emmy-vllm endpoint). Preservation-policy enforcement happens by passing `customInstructions` that include emmy's D-14 preservation directive text (from `prompts/compact.md`).

### D-31 — /compact slash command
Add `pi.registerCommand("compact", { handler })` in `packages/emmy-ux/src/pi-emmy-extension.ts`. Handler signature: `async (args, ctx) => ctx.compact({ customInstructions: args || undefined })`. Manual trigger; user can force compaction at any moment (useful when running many file reads before asking for a summary). Works even below the D-11 soft threshold.

### D-32 — /clear slash command
Add `pi.registerCommand("clear", { handler })` in `packages/emmy-ux/src/pi-emmy-extension.ts`. Handler uses `ctx.newSession({})` from pi 0.68 `ExtensionCommandContext` (verified at `dist/core/extensions/types.d.ts:236-241`). Emmy's SP_OK canary + 3-layer prompt assembly re-run automatically on the new session because session.ts's boot hooks fire on each AgentSession creation. `/clear` is the only escape from an over-full session state short of killing pi-emmy.

### D-33 — SearxNG topology
Self-hosted SearxNG Docker Compose stack at `observability/searxng/docker-compose.yaml`, sibling of `observability/langfuse/docker-compose.yaml`. Single service (plus Redis for the search cache — use the SAME `redis` container Langfuse already runs via `external: true`, no new Redis). Digest-pinned image per D-09 (Phase 3) air-gap discipline. Bind `127.0.0.1:8888:8080` loopback-only. Config `observability/searxng/settings.yml` enables these engines:
- `google` (primary, preferred)
- `duckduckgo`, `brave`, `bing` (automatic fallbacks — SearxNG rotates on rate-limit or timeout, which is the exact problem we're solving)

Setup script `scripts/start_searxng.sh` mirrors `start_observability.sh`: openssl-generated secret-key, health-gate with 60s timeout, instructions banner at end. `scripts/stop_searxng.sh` mirrors `stop_observability.sh` with optional `--volumes` flag.

### D-34 — web_search tool
New tool at `packages/emmy-tools/src/web-search.ts`:
```
web_search(query: string, max_results?: number = 10): Promise<SearchResult[]>
```
Where `SearchResult = { title: string; url: string; snippet: string; engine: string }`. Implementation: HTTP GET to `http://127.0.0.1:8888/search?q=<urlencoded>&format=json&safesearch=0&max_results=<n>`. On HTTP 5xx or timeout (10s default), returns empty array + emits `tool.web_search.fallback` telemetry event; does NOT throw (pi agent continues). On kill-switch (EMMY_WEB_SEARCH=off OR EMMY_TELEMETRY=off), the tool is unregistered — pi never advertises it to the model.

### D-35 — web_fetch returned-URL bypass
`packages/emmy-tools/src/web-fetch-allowlist.ts` grows a sibling concept to the hostname allowlist: a rolling in-memory `recentSearchUrls: Set<string>` populated by `web_search` on every successful call. URL is fetchable IFF `hostname ∈ allowlist` OR `url ∈ recentSearchUrls` (exact URL match, not hostname). Entries expire after 5 minutes (configurable via `tools.web_fetch.search_bypass_ttl_ms` in harness.yaml, default 300000). Plan 03.1-02 also preserves v3.1's hostname allowlist with the same 5 default hosts as v3, so existing allowlisted doc-host flows continue to work.

### D-36 — Badge rename (three-state)
`packages/emmy-ux/src/offline-badge.ts` grows from two states to three:
- `[emmy] OFFLINE OK` (green) — SearxNG is down, disabled, or unreachable; web_search unavailable; fully air-gapped posture
- `[emmy] LOCAL LLM · WEB` (yellow, new ANSI 33) — SearxNG is up and healthy; web_search live; LLM inference still local-only
- `[emmy] CLOUD INFERENCE` (red) — reserved for actual thesis violation; non-loopback inference endpoint detected

The existing "NETWORK USED" label is DEPRECATED; boot-time tool audit now distinguishes benign egress (SearxNG) from thesis-breaking egress (non-loopback vLLM/inference). The red state should NEVER fire with current code — it's structurally reserved so that Phase 4 (when multi-model routing arrives) and Phase 6 (speculative decoding proxies) have a natural home for detection.

## Existing Code Insights

- **`emmyCompactionTrigger` injection seam is already correct.** Plan 03-03 left `engine.summarize` as an injectable function that throws "not configured" by default. Plan 03.1-01 replaces that with the live wiring — NOT by implementing a raw HTTP round-trip, but by having the turn_start handler in `pi-emmy-extension.ts` call `ctx.compact({customInstructions})` DIRECTLY. This is a cleaner architecture than injecting a summarize function because pi's ctx.compact already knows about the active Model and apiKey. The `emmyCompactionTrigger` body keeps its role as the D-14 preservation classifier + D-11 soft-threshold + D-12 hard-ceiling guardrails; when the trigger determines compaction SHOULD fire, it now emits a structured directive (a preparation object) rather than calling summarize itself, and the turn_start handler reads the directive + calls ctx.compact. Net behavior: same soft-threshold semantics, same preservation semantics, now running against pi's native compaction engine via the active Model.
- **Slash command seam:** `pi.registerCommand("compact", ...)` + `pi.registerCommand("clear", ...)` inside `createEmmyExtension` — right next to the existing `pi.registerShortcut("shift+ctrl+up/down", ...)` (Plan 03-08 pattern). Both commands take `ExtensionCommandContext` which has `.compact()`, `.newSession()`, `.waitForIdle()`, `.abort()`.
- **SearxNG JSON API is plain JSON over HTTP GET.** No auth headers, no rate-limiting from localhost, no websockets. Request shape: `GET /search?q=<urlencoded>&format=json&safesearch=0`. Response: `{query, results: [{title, url, content, engine, ...}], answers, suggestions, ...}`. The new `web_search` tool can be ~60 lines of TS (fetch + decode + map) — smaller than web_fetch.
- **Badge state-machine is already module-level** (Plan 03-06). `setInitialAudit`, `flipToViolation`, `bindBadge`, `_lastResult` — Plan 03.1-02 extends the enum from `{green, red}` to `{green, yellow, red}` and adds `flipToYellow(reason)` / `flipToGreen(reason)` for SearxNG up/down transitions. The existing `flipToViolation` stays reserved for the D-36 red state (non-loopback inference).
- **Profile hash `--write`** (Phase 2 Plan 02-07 pattern) — `uv run emmy profile hash --write profiles/qwen3.6-35b-a3b/v3.1` computes + writes to `profile.yaml`. Strips any `KNOWN-STALE` comments. v3 stays byte-identical (NOT modified).

## Deferred Ideas (confirmed out-of-scope for Phase 3.1)

- **Per-user SearxNG cookie settings** — SearxNG supports per-session UI preferences. Emmy is single-user and uses JSON API only; no cookies. Revisit only if we add a web UI.
- **SearxNG engine weighting / category filters** — default engine priority (google > ddg > brave > bing) is hardcoded in settings.yml; emmy doesn't expose knobs. Revisit if certain search types (code, academic) need dedicated engines.
- **Result scoring / de-duplication inside emmy** — SearxNG already de-duplicates cross-engine hits server-side. Emmy treats SearxNG's JSON output as authoritative.
- **Cache warm-up / pre-fetch** — no; the agent's search pattern is unpredictable.
