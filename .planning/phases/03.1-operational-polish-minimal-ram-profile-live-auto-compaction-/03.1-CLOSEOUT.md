## Walkthrough-discovered lessons captured

**Live walkthrough is load-bearing for integration defects.** Plans 03-05, 03-08, and 03.1-02 each shipped unit-test-green and failed live. In every case the unit tests mocked the integration layer where the bug lived. Phase 3.1 Plan 03.1-02 specifically discovered three layers that need fixing when adding a new tool:

1. **profile-loader.ts** — must parse the new YAML block from `harness.yaml`
2. **session.ts prompt-assembly** — the hardcoded `toolDefsText` must include the tool so the LLM knows it exists
3. **profile's `prompts/tool_descriptions.md`** — descriptive layer (nice-to-have; prompt-assembly is the authoritative source)

Future tool-addition plans (Phase 4 will add new tools per Gemma 4) should include ALL THREE in `files_modified` explicitly; don't rely on unit tests to catch gaps.

## Thesis revision summary

The Phase 1 / 2 / 3 documents said "No cloud dependency anywhere in the loop". Phase 3.1 refines this into a more honest two-axis statement:

- **No cloud INFERENCE** — the LLM is 100% local on emmy-serve. Hard invariant. Air-gap-tested via `ci_verify_phase3`. Every model weight stays on DGX Spark.
- **Local-first EGRESS** — SearxNG is the single authorized outbound component. Loopback from the harness's POV (harness only talks to `127.0.0.1:8888`). SearxNG itself makes outbound queries to search engines. Auditable (`docker logs searxng-searxng-1`), disableable (`EMMY_WEB_SEARCH=off` or stop the container), observable (every query logged as `tool.web_search` event). `ci_verify_research_egress` permits SearxNG outbound while denying 12 cloud-inference endpoints (OpenAI, Anthropic, Google Generative Language, AWS Bedrock, etc.).

The OFFLINE OK badge is now honest:
- `OFFLINE OK` (green) — only when SearxNG is DOWN/disabled
- `LOCAL LLM · WEB` (yellow) — SearxNG up, web_search active
- `CLOUD INFERENCE` (red) — reserved; would fire only on a genuine thesis violation
