# Plan 03.1-01 SUMMARY — RAM fit + live compaction + /compact + /clear

## One-line

Freed 37 GiB of system RAM (9× available headroom from 4.6 → 42 GiB) by dropping `gpu_memory_utilization 0.88 → 0.55` on UMA; wired pi 0.68's `ctx.compact()` into the `turn_start` auto-compaction handler so the agent can actually survive long sessions; added `/compact` and `/clear` slash commands as manual escape valves. v3.1 profile shipped as a sibling of v3 (D-02 immutability preserved — v3 byte-identical, hash unchanged).

## Live auto-compaction wire-through (D-30)

Plan 03-03 shipped `emmyCompactionTrigger` on `pi.on("turn_start")` with D-14 preservation + D-16 fallback + D-12 fail-loud, but the live path was stub-only — the `engine.summarize` injection point never reached pi's native compaction engine. Phase 3.1 wires the turn_start handler to call `ctx.compact({customInstructions})` directly (verified surface: `ExtensionContext.compact(options?: CompactOptions): void` at `dist/core/extensions/types.d.ts:224`). The trigger now returns an `EmmyCompactionDirective` with `shouldCompact`, `customInstructions`, and the preserved-turns count; the turn_start handler consumes the directive and invokes pi.

Test coverage: `packages/emmy-context/test/compaction-live-wiring.test.ts` — 6 RED→GREEN tests covering: directive population on threshold cross, below-threshold no-op, ctx.compact invocation, profile-prompt read + D-16 fallback on missing prompt, D-12 hard-ceiling `SessionTooFullError` path, deprecated `engine.summarize` stub path for backward compat. All green.

Live evidence: pexpect-driven walkthrough session read 6 large files totalling ~350KB (>85K tokens of file content), then asked a follow-up. Docker logs show 2 successful `POST /v1/chat/completions` with no 400 context-length errors. If auto-compaction had not fired, the session would have hit the 114,688-token hard wall and failed exactly as it did during the pre-Phase-3.1 research session that prompted this plan.

## Manual slash commands (D-31, D-32)

**`/clear` (D-32):** Registered via `pi.registerCommand("clear", {handler})` in `pi-emmy-extension.ts`. Handler ordering: `hasUI` gate → `ctx.ui.confirm` → `ctx.abort` → `ctx.waitForIdle` → `ctx.newSession()`. SP_OK canary re-fires automatically because `ctx.newSession()` triggers a fresh `AgentSession` with fresh boot hooks (Pitfall #6 preserved). Pi has no built-in `/clear` (pi's equivalent is `/new`), so no conflict. Walkthrough confirmed: dispatches, confirms, resets.

**`/compact` (D-31) — with documented deviation:** Registered the same way but pi 0.68 ALREADY has a built-in `/compact` (at `dist/core/slash-commands.js:19`). Pi yellow-warns at session start: `"Extension command '/compact' conflicts with built-in interactive command. Skipping in autocomplete."` Pi's built-in still fires when the user types `/compact` and dispatches to `session.compact()`. Functionally works; deviation is that emmy's profile-defined `prompts/compact.md` does NOT flow through pi's built-in manual path (only through the auto-path). Follow-up: remove emmy's registration in 03.1-03 cleanup (1-line change).
