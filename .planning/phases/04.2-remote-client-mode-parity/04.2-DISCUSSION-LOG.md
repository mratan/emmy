## Sidecar API surface + lifecycle

### Q1 — /stop in-flight behavior

| Option | Description | Selected |
|--------|-------------|----------|
| Graceful drain w/ 30s timeout | Stop accepting new requests, let in-flight finish, then SIGTERM after 30s grace; SIGKILL if still alive. Sidecar reports `draining N requests` progress event. | ✓ |
| Hard kill immediately | SIGTERM right away regardless of in-flight. Faster RAM reclaim. | |
| Configurable, default graceful | POST /stop accepts `{force, timeout_s}`. CLI defaults to graceful; advanced users pass `--force`. | |

**User's choice:** Graceful drain w/ 30s timeout (Recommended)
**Notes:** Protects multi-minute generations on the daily-driver. → D-01

### Q2 — /start <profile> semantics

| Option | Description | Selected |
|--------|-------------|----------|
| Any profile + idempotent same-profile | `/start <name>[@<variant>]` accepts any profile; idempotent if same is up; runs swap orchestrator if different. Single mental model. | ✓ |
| Only configured default profile | `/start` takes no args; same env-default every time. `/profile` is the only switcher. | |
| Any profile, error if already up | `/start` only valid when stopped; 409 Conflict if running. | |

**User's choice:** Any profile + idempotent (Recommended) — with clarification request: "confirm that variant means the v number (eg. v1)"
**Notes:** Variant clarified via `routes-loader.ts:14`. Variant = the directory name under `profiles/<profileId>/<variant>/`, packing version + optional role suffix together (`v3.1-default`, `v3.1-reason`, `v1.1`). Same syntax `/profile` already accepts. → D-02

### Q3 — /status delivery model

| Option | Description | Selected |
|--------|-------------|----------|
| GET /status only (poll) | Mac client polls every N seconds. SSE only for /start, /stop, /profile progress. Smaller surface. | ✓ |
| GET /status + SSE /events (push) | Long-lived SSE for status changes. More responsive UI but extra failure modes (reconnect). | |

**User's choice:** GET /status only (Recommended)
**Notes:** Footer poller in `packages/emmy-ux/src/metrics-poller.ts` becomes the consumer. → D-03

## Local-mode parity strategy

### Q4 — Single-path vs dual-path dispatcher

| Option | Description | Selected |
|--------|-------------|----------|
| Dual path: spawn locally, HTTP remotely | When `EMMY_REMOTE_CLIENT` unset → spawn(uv,…) byte-stable. =1 → HTTP+SSE. Two paths, zero local regression risk. | ✓ |
| Single path: always sidecar HTTP | Local mode runs sidecar on loopback; spawn path replaced wholesale. Cleaner, more risk for same end-state. | |
| Single path with feature flag rollback | Sidecar default; spawn behind `EMMY_LEGACY_SWAP=1` for one phase. Hedges. | |

**User's choice:** Dual path (Recommended)
**Notes:** Phase 4 just stabilized spawn-orchestrator with operator-gated thermal evidence. ~80 LOC adapter, hot path unchanged. → D-04

## Sidecar process lifecycle on Spark

### Q5 — Always-on mechanism

| Option | Description | Selected |
|--------|-------------|----------|
| systemd user unit | `emmy-serve/systemd/emmy-sidecar.service`, `--user` enabled, `loginctl enable-linger $USER` for boot survival. Auto-restart. | ✓ |
| `scripts/start_sidecar.sh` + manual restart | One-shot script. No systemd dep. No reboot survival, no auto-restart. | |
| Docker Compose alongside Langfuse + SearxNG | Docker-in-Docker / socket-bind for spawning vLLM container is awkward. | |
| tmux/screen helper | Detached tmux session. Survives shell exit, not reboot. Workaround feel. | |

**User's choice:** systemd user unit (Recommended)
**Notes:** Standard Linux pattern; works headless; survives reboot. → D-05

### Q6 — Python web framework

| Option | Description | Selected |
|--------|-------------|----------|
| FastAPI + uvicorn | Async, SSE via StreamingResponse, pydantic-native (already pinned). ~30 LOC for 4 endpoints. | ✓ |
| aiohttp | Lighter than FastAPI. SSE manual; no pydantic integration. | |
| bare http.server | Zero deps. Synchronous, no native SSE, hand-rolled. Maintenance burden. | |

**User's choice:** FastAPI + uvicorn (Recommended)
**Notes:** Adds `fastapi>=0.110`, `uvicorn[standard]>=0.27` to pyproject.toml. Module = `emmy_serve/swap/controller.py`. → D-06

## /status payload + test posture

### Q7 — /status payload

| Option | Description | Selected |
|--------|-------------|----------|
| Rich (vLLM up + profile + KV% + GPU temp + digest + in-flight) | Single GET with everything Mac footer + debugger want. Sidecar polls vLLM /metrics + nvidia-smi internally. | ✓ |
| Minimal (state + profile only) | Mac client polls separately — but vLLM /metrics + nvidia-smi don't work over Tailscale, so Mac footer would be degraded. | |
| Minimal + GET /metrics passthrough | Two endpoints. Cleaner separation but more surface. | |

**User's choice:** Rich (Recommended)
**Notes:** Sidecar caches `nvidia-smi` for 1s to avoid stampedes. Optional fields are `null` when not computable (e.g. `kv_used_pct` when `vllm_up=false`). → D-07

### Q8 — Test posture / acceptance bar

| Option | Description | Selected |
|--------|-------------|----------|
| Unit + integration + operator-gated E2E | Sidecar endpoint unit tests, HTTP-runner unit tests, real-sidecar (no vLLM) integration tests, operator-gated tailnet SC walkthrough. | ✓ |
| Unit + integration only (no E2E) | Skip operator-gated walkthrough. Risk of broken-in-practice phase. | |
| Unit only (mock everything) | Sidecar bugs surface only in production. Reject for daily-driver path. | |

**User's choice:** Unit + integration + operator-gated E2E (Recommended)
**Notes:** Mirrors Phase 4 evidence pattern; SC-1/2/3 phase4.2 with resume signals → Done† promote model. → D-08
