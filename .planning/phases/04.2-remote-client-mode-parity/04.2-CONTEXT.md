# Phase 04.2: Remote-client mode parity - Context

**Gathered:** 2026-04-25
**Status:** Ready for planning

<domain>
## Phase Boundary

Make `/profile`, `/start`, `/stop`, `/status`, and `web_search` work from a Mac/laptop client over Tailscale. Today these all hardcode `127.0.0.1` and break the moment the harness runs anywhere except the Spark host.

Two structural pieces:

1. **Spark side** — an always-on Python sidecar (`emmy_serve.swap.controller`) that owns vLLM lifecycle. The sidecar exposes an HTTP+SSE control plane and lets the user `/stop` to reclaim RAM, `/start <profile>[@<variant>]` to wake, `/profile` to swap, and `/status` to inspect. vLLM stays on-demand; only the sidecar is always-on.
2. **Mac side** — `profile-swap-runner.ts` becomes a dispatcher that, when `EMMY_REMOTE_CLIENT=1`, routes to the sidecar over HTTP+SSE instead of `spawn("uv", "-m", "emmy_serve.swap.orchestrator", …)`. `web-search.ts` honors `EMMY_SEARXNG_URL` so the Mac can hit the Spark-hosted SearxNG via Tailscale Serve. `install-client.sh` is extended to provision the two tailscale serve routes. **Local mode is byte-stable** — the spawn path is preserved when `EMMY_REMOTE_CLIENT` is unset.

Air-gap posture (D-33 LOCKED): in local mode, SearxNG remains loopback-only; the `EMMY_SEARXNG_URL` override is the documented escape hatch for remote-client posture, not a profile change. `ci_verify_phase3` (STRICT — zero outbound) keeps gating the inference loop. `ci_verify_research_egress` (PERMISSIVE) is unchanged.

</domain>

<decisions>
## Implementation Decisions

### Sidecar API surface

- **D-01:** `/stop` does graceful drain with a 30-second grace period. Sidecar stops accepting new requests, lets in-flight finish, emits a `draining N requests` SSE event, then SIGTERM vLLM at the deadline; SIGKILL if still alive after another 5s. Protects multi-minute generations on the daily-driver. **Why:** the Spark vLLM container can hold a 60+ second tool-call generation; hard-killing mid-stream would corrupt the user's session, especially over a Tailscale hop where latency makes the user less likely to know exactly when they pressed `/stop`.
- **D-02:** `/start <profileId>[@<variant>]` accepts the **exact same `<profileId>@<variant>` syntax** that the existing `/profile` slash command accepts (see `packages/emmy-ux/src/slash-commands.ts:253` and `packages/emmy-ux/src/routes-loader.ts:14`). The variant string is the directory name under `profiles/<profileId>/<variant>/` and packs version + optional role suffix together (e.g. `qwen3.6-35b-a3b@v3.1-default`, `qwen3.6-27b@v1.1`). Behavior:
  - If vLLM is **stopped**, sidecar starts the requested bundle (existing `emmy_serve.swap.orchestrator --to <variant>` path, but without the `--from` since nothing is running).
  - If vLLM is **already running on the same variant**, return `{state: "ready", phase: "ready"}` immediately (idempotent).
  - If vLLM is **already running on a different variant**, sidecar runs the existing swap orchestrator internally (same `--from X --to Y --port N` invocation) and emits the **D-02 LOCKED Phase-4 progress sequence** (`stopping vLLM` → `loading weights` 0/50/90 → `warmup` → `ready`) over SSE. Reuses `emmy_serve/swap/orchestrator.py` end-to-end — no second progress contract.
- **D-03:** `/status` is a **single GET (poll-only)**. SSE is reserved for `/start`, `/stop`, `/profile` progress streams. The Mac footer poller (`packages/emmy-ux/src/metrics-poller.ts`) becomes the consumer of `/status` and will replace today's direct `nvidia-smi` + vLLM `/metrics` calls in remote mode. Local mode keeps the existing direct calls (or optionally polls the loopback sidecar — see D-09).

### Local-mode parity

- **D-04:** **Dual-path dispatcher** in `packages/emmy-ux/src/profile-swap-runner.ts`:
  - `EMMY_REMOTE_CLIENT` unset / `=0` → today's `spawnFn("uv", ["run", "python", "-m", "emmy_serve.swap.orchestrator", …])` path is **byte-stable**. Zero daily-driver regression.
  - `EMMY_REMOTE_CLIENT=1` → dispatcher routes through HTTP+SSE to the sidecar at `${EMMY_SERVE_URL}/profile/swap` (POST starts the swap, response is `text/event-stream` SSE).
  - Both paths emit identical `{phase, pct?}` records to `opts.onProgress` and the same exit-6 envelope (`{rolled_back, rollback_succeeded?}`). Downstream UI (`harness-swap.ts`, the inline progress meter) is unchanged.
  - **Why:** Phase 4 just stabilized the spawn-orchestrator path with operator-gated thermal evidence. A wholesale rewrite to single-path-via-sidecar would require full daily-driver re-validation. The dual path costs ~80 LOC of HTTP-mode adapter and zero LOC change to the hot path.

### Sidecar process lifecycle

- **D-05:** Sidecar runs as a **systemd user unit**, not Docker, not tmux, not a one-shot script.
  - Service file: `emmy_serve/systemd/emmy-sidecar.service` (in-repo, copied to `~/.config/systemd/user/` by an install task).
  - Enable: `systemctl --user enable --now emmy-sidecar`.
  - One-time op step: `loginctl enable-linger $USER` so it survives reboot without an active SSH session.
  - `Restart=on-failure`, `RestartSec=5`, `Environment=PYTHONUNBUFFERED=1` and the standard emmy env vars (HF_HOME, VLLM_NO_USAGE_STATS=1, VLLM_LOAD_FORMAT=fastsafetensors).
  - Documented in `docs/runbook.md` next to the existing emmy-serve operations and in `scripts/start_emmy.sh` as a step the user runs once during initial setup.
- **D-06:** Sidecar is **FastAPI + uvicorn**.
  - Add `fastapi>=0.110` and `uvicorn[standard]>=0.27` to `pyproject.toml` `[project.dependencies]`.
  - Module: `emmy_serve/swap/controller.py` (sibling to existing `orchestrator.py`, `preflight.py`, `progress.py`, `rollback.py`).
  - Bound to `0.0.0.0:8003` so Tailscale Serve can re-proxy. Loopback access for local mode is the same socket.
  - Pydantic models for request/response schemas (we already pin pydantic>=2.6).
  - SSE via FastAPI's `StreamingResponse(media_type="text/event-stream")`.

### /status payload

- **D-07:** `/status` returns rich JSON:
  ```json
  {
    "state": "stopped" | "starting" | "ready" | "swapping" | "draining" | "error",
    "profile_id": "qwen3.6-35b-a3b" | null,
    "profile_variant": "v3.1-default" | null,
    "profile_hash": "<64-hex>" | null,
    "vllm_up": true | false,
    "vllm_pid": 12345 | null,
    "container_digest": "sha256:..." | null,
    "kv_used_pct": 0.34 | null,
    "gpu_temp_c": 64.2 | null,
    "in_flight": 2 | null,
    "last_error": null | { "ts": "...", "msg": "..." }
  }
  ```
  - Sidecar computes optional fields by polling vLLM `/metrics` (loopback) and `nvidia-smi --query-gpu=temperature.gpu --format=csv,noheader` once per /status call (cache for 1s to avoid stampedes).
  - Fields are `null` when not computable (e.g. `kv_used_pct` is null when `vllm_up=false`).
  - State transitions are sidecar-internal: `stopped → starting → ready → swapping → ready`, `ready → draining → stopped`, any → `error`. Recovery from `error` requires `/start`.
- **D-09:** Mac footer poller (`packages/emmy-ux/src/metrics-poller.ts`) is extended with a remote-mode branch: when `EMMY_REMOTE_CLIENT=1`, poll `${EMMY_SERVE_URL}/status` instead of calling `nvidia-smi` and vLLM `/metrics` directly. Local mode keeps the direct-call path. Same dual-path discipline as D-04.

### Test posture

- **D-08:** Phase 04.2 acceptance = **unit + integration + operator-gated E2E**:
  1. **Unit tests** for each sidecar endpoint (`tests/unit/test_sidecar_*.py`) — mock the underlying `emmy_serve.swap.orchestrator` invocation, assert HTTP and SSE event shapes.
  2. **Unit tests** for HTTP-mode `profile-swap-runner.ts` (`packages/emmy-ux/test/profile-swap-runner.http.test.ts`) — fetch + SSE mocked via DI (mirroring the existing `spawnFn` injection pattern).
  3. **Integration tests** that boot the real sidecar without a real vLLM (`tests/integration/test_sidecar_no_vllm.py`) — the sidecar should report `state: "stopped", vllm_up: false` and reject `/profile` (since there's nothing to swap). Marked `@pytest.mark.integration`, opt-in via `--run-integration`.
  4. **Operator-gated SC walkthrough** on the actual tailnet (Mac → Spark) — recorded as `SC-1 phase4.2: Mac client one-shot through Tailscale`, `SC-2 phase4.2: /start /stop /status round-trip`, `SC-3 phase4.2: /profile swap from Mac`. Each gets a resume signal (e.g. `sc1 phase4.2 green`) so the phase can close as Done† and promote on operator confirmation, mirroring Phase 4.

### Claude's Discretion

The following are not user decisions — Claude/researcher/planner own them:

- Internal sidecar HTTP route names: `POST /start`, `POST /stop`, `GET /status`, `POST /profile` (or `/profile/swap`). Researcher picks the exact paths; D-01–D-03 fix the *behavior*.
- SSE event field names beyond `phase` and `pct` (e.g. `ts`, `level`, `details`). The wire-level JSON-per-line shape is fixed by C-06; SSE wrapping is up to the planner.
- Sidecar ↔ orchestrator IPC: subprocess + parse stdout (matches today's path), or import `orchestrator.py` as a module and call `run_swap()` directly. Planner decides based on which is cleaner for FastAPI.
- Backoff/retry policy in HTTP-mode `profile-swap-runner.ts` (timeout, reconnect on transient SSE drop, etc.).
- Exact metrics-poller polling cadence in remote mode (probably 2s like today's `nvidia-smi` cadence).
- Whether `install-client.sh` writes `EMMY_REMOTE_CLIENT=1` and `EMMY_SERVE_URL=…` into the wrapper script, or sources a generated `.envrc`.
- systemd unit ExecStart wrapper (uv vs venv vs direct python).

## Existing Code Insights

### Reusable Assets

- **`emmy_serve/swap/orchestrator.py`** — already does start/stop/swap with the exact JSON-per-line progress contract this phase needs. Sidecar wraps it; does not duplicate it.
- **`emmy_serve/swap/preflight.py`, `rollback.py`, `progress.py`** — supplementary primitives the orchestrator already uses; sidecar reaches them through orchestrator, not directly.
- **`packages/emmy-ux/src/profile-swap-runner.ts:72`** — `runSwapAndStreamProgress` already has DI for `spawnFn`. The HTTP-mode branch reuses the same `(phase, pct?)` callback and exit-envelope shape. ~80 LOC of new HTTP/SSE code beside it.
- **`packages/emmy-ux/src/slash-commands.ts:168-280`** — `/profile` registration via `pi.registerCommand` is the template for `/start`, `/stop`, `/status`.
- **`packages/emmy-ux/src/metrics-poller.ts`** — already polls on a 2s cadence and renders the footer; D-09 just adds a remote branch.
- **`packages/emmy-ux/src/vllm-metrics.ts`** — vLLM Prometheus parser the sidecar imports server-side (Python equivalent in `emmy_serve/diagnostics/`? Check during planning).
- **`scripts/install-client.sh`** — already detects Tailscale, finds Spark on the tailnet, writes a wrapper to `~/.local/bin/emmy`. Just needs two new `EMMY_REMOTE_CLIENT=1` + `EMMY_SERVE_URL=…` lines and the second/third `tailscale serve` reminders in the Spark-side block.

### Established Patterns

- **DI for testability** — every TS module that touches a side-effecting boundary (spawn, fetch, fs) accepts a mock parameter. HTTP-mode swap runner inherits this discipline (mock `fetch` and SSE EventSource).
- **JSON-per-line over stdout** — Phase 4's contract; SSE just wraps it in `data: ...\n\n` framing. No semantic change.
- **DI-test integration test** — `@pytest.mark.integration` opt-in for tests that need a real sidecar/vLLM/network.
- **Slash command registration** — `pi.registerCommand(name, { description, handler })` in `slash-commands.ts`, registered from `pi-emmy-extension.ts`.
- **Three-environment-flag pattern** — emmy already uses many env vars (`EMMY_PROFILE_ROOT`, `EMMY_SKIP_PROFILE_VALIDATE`, `EMMY_WEB_SEARCH`, `EMMY_BASE_URL`, `PI_SKIP_VERSION_CHECK`). Adding `EMMY_REMOTE_CLIENT`, `EMMY_SERVE_URL`, `EMMY_SEARXNG_URL` is consistent.
- **Profile validate at boot** — local mode runs `emmy profile validate` at startup. Remote-client install-client.sh already sets `EMMY_SKIP_PROFILE_VALIDATE=1` (the Mac doesn't have profile bundles by default). No change needed.

### Integration Points

- **Sidecar↔harness:** `${EMMY_SERVE_URL}/start`, `/stop`, `/status`, `/profile/swap` over HTTP+SSE. `EMMY_SERVE_URL` defaults to `http://127.0.0.1:8003` if unset; can be `https://<spark>.<tailnet>.ts.net` (when reusing the existing `:8002` Tailscale Serve route on a path prefix) OR `https://<spark>.<tailnet>.ts.net:<separate-port>` (with a second tailscale serve binding). Researcher picks the cleaner of these two.
- **Sidecar↔vLLM:** loopback `http://127.0.0.1:8002/v1/models`, `/metrics`. Sidecar reuses the existing port discipline.
- **Sidecar↔orchestrator:** subprocess (`subprocess.Popen("uv", ["run", "python", "-m", "emmy_serve.swap.orchestrator", …])`) OR module import (`from emmy_serve.swap import orchestrator; orchestrator.run_swap(…)`). Planner picks; subprocess matches today's TS-side pattern, module import is cleaner Python.
- **Tailscale Serve:** Spark side runs three `tailscale serve --bg` invocations (vLLM 8002, sidecar 8003, SearxNG 8888). MagicDNS HTTPS termination handled by Tailscale; sidecar speaks plain HTTP behind it.
- **systemd:** user-level unit (`~/.config/systemd/user/emmy-sidecar.service`), enabled with `--user`; lingering enabled via `loginctl enable-linger`.

The badge's logic ALREADY supports this — D-09 (poller remote branch) just feeds it the right inputs.

<deferred>
## Deferred Ideas

These came up implicitly but belong in other phases:

- **Telemetry/Langfuse over Tailscale** — README briefly mentions wiring Langfuse via a third tailscale serve route. Out of scope for this phase; keep `LANGFUSE_HOST=http://localhost:3000` semantics on Spark, Mac sessions write JSONL locally for now. Revisit when telemetry-from-remote-client is a felt need.
- **Token-based auth** — would matter for multi-user or non-Tailscale deployment. Trust-on-tailnet (C-05) is sufficient for the single-user thesis. Backlog if/when emmy goes multi-tenant.
- **Cross-machine profile bundles** — Mac currently sets `EMMY_SKIP_PROFILE_VALIDATE=1` because it lacks bundles. Could rsync bundles or expose a sidecar `/profiles` endpoint that streams hashes. Not needed; variant validation happens server-side at swap time.
- **Multi-Spark tailnet routing** — tailnet could host multiple Sparks; client picks one via env var. Single-Spark assumption is fine for v1.
- **Sidecar OpenAPI / introspection endpoint** — FastAPI gives `/docs` for free; deciding whether to expose it through Tailscale Serve is a security-posture call deferred to runbook discussion.
- **`/restart` shortcut** — `/stop` + `/start` is two commands; some users may want a single `/restart`. Easy to add later as a thin wrapper on the sidecar; not v1.
- **Telemetry of sidecar-itself** — sidecar's own request metrics (latency, in-flight). Useful for debugging Tailscale-Serve hiccups. Add when first felt.
