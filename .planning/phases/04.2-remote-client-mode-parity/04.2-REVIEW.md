# Phase 04.2: Code Review Report

**Reviewed:** 2026-04-25T00:00:00Z
**Depth:** standard

## Summary

Phase 04.2 adds a FastAPI sidecar (`emmy_serve/swap/controller.py`) exposing HTTP+SSE lifecycle control, a Mac-side TypeScript dispatcher with HTTP+SSE clients, slash commands for `/start`, `/stop`, and `/status`, an installer script, and an env-driven SearxNG URL override in `web-search.ts`.

The subprocess boundary is clean: `asyncio.create_subprocess_exec` is used throughout `orchestrator_runner.py`; no `shell=True` path exists anywhere in Phase 04.2 code. The SSE consumption layer correctly uses the eventsource-parser v3 `createParser({ onEvent: fn })` API in all three TS modules. The `EMMY_REMOTE_CLIENT` dispatcher branch in `profile-swap-runner.ts` is a thin 4-line additive block that leaves the existing spawn body byte-identical. D-33 LOCKED is preserved: the `_resolveDefaultBaseUrl()` getter in `web-search.ts` defaults to `http://127.0.0.1:8888` when `EMMY_SEARXNG_URL` is unset. The air-gap smoke validators do not make network calls during test execution.

Three warnings and three info items are noted below.

## Warnings

### WR-01: `/profile/swap` endpoint missing path-traversal validation present on `/start`

**File:** `emmy_serve/swap/controller.py:469-546`

**Issue:** The `/start` handler calls `_validate_profile_id_and_variant()` which rejects `"/"` or `".."` in `profile_id` and `variant` fields before invoking the orchestrator subprocess. The `/profile/swap` handler (`profile_swap`) receives `req.from_` and `req.to` as free-form strings and passes them directly to `run_orchestrator_subprocess(from_path=req.from_, to_path=req.to)` with no analogous check. An attacker with tailnet access could supply `"../../some/arbitrary/path"` as `req.to`, which would be forwarded as the `--to` argument to the orchestrator subprocess.

Shell injection is not possible because `create_subprocess_exec` is used (no shell expansion). The orchestrator's D-05 preflight validation (`run_preflight`) would reject paths that don't contain a valid `serving.yaml`, providing a backstop. However, the missing guard breaks the stated security invariant at line 23 of the module docstring ("Path traversal guards on /start") and allows the orchestrator to attempt file-system access on arbitrary paths before the preflight fires.

The module docstring at line 23 explicitly documents the T-04.2-S3 mitigation as applying only to `/start`, omitting `/profile/swap`. The trust boundary is C-05 (tailnet-only), but defense-in-depth still applies to the sidecar's own endpoints.

**Fix:**
```python
@app.post("/profile/swap")
async def profile_swap(req: SwapRequest, request: Request) -> EventSourceResponse:
    # Add path-traversal guard before state check (same pattern as /start).
    for field_name, field_val in [("from", req.from_), ("to", req.to)]:
        if "/" in field_val or ".." in field_val:
            raise HTTPException(
                status_code=400,
                detail=f"invalid {field_name} path (no '/' or '..')",
            )

    if state.state != SidecarState.READY:
        ...
```

### WR-02: Client disconnect during `/start` or `/profile/swap` leaves orchestrator subprocess running with no cleanup, causing state machine lock in ERROR until next `/start`

**File:** `emmy_serve/swap/controller.py:307-351` and `emmy_serve/swap/orchestrator_runner.py:94-99`

**Issue:** When a Mac client disconnects mid-SSE stream during `/start` or `/profile/swap`, the `event_generator` breaks out of the `async for rec in run_orchestrator_subprocess(...)` loop (lines 328-329 and 517-518). Two consequences follow:

1. **State machine left in ERROR.** After the break, `exit_rc` is `None` (the `_internal_exit` sentinel was never seen). The `else` branch at line 337 fires: `_record_error(f"orchestrator exit {exit_rc}")` and `await state.transition_to(SidecarState.ERROR)`. The sidecar is now in ERROR state even though the orchestrator subprocess is still actively running (doing docker operations).

2. **Subprocess not killed.** `orchestrator_runner.py`'s `finally:` block only calls `rc = await proc.wait()` (line 97) — it never calls `proc.kill()` or `proc.terminate()`. The subprocess continues running (potentially for the full warmup timeout of 900 seconds) while the state machine reports ERROR. During this window, a new `/start` call (required to recover from ERROR) could race with the still-running subprocess, causing two orchestrator instances to contend over the same Docker container name.

**Fix for `orchestrator_runner.py`:** Kill the subprocess on generator close before waiting:
```python
finally:
    if proc.returncode is None:
        try:
            proc.kill()
        except ProcessLookupError:
            pass
    rc = await proc.wait()
```

**Fix for `controller.py` post-break state transition:** After a disconnect-triggered break, leave state in STARTING/SWAPPING (not ERROR) so the sidecar FSM reflects "in progress but client lost" rather than "failed":
```python
if await request.is_disconnected():
    # Do not immediately transition to ERROR — subprocess may still succeed.
    # Leave state as-is; next /start will find state != {STOPPED,READY} and 409.
    break
```
Alternatively, add a `disconnect_signalled` flag and skip the ERROR transition on break-without-exit-code. The exact strategy is a design call, but the current behavior (ERROR + orphaned subprocess) is the worst combination.

### WR-03: `new URL("/search", cfg.baseUrl)` in `web-search.ts` is outside the try/catch; `EMMY_SEARXNG_URL=""` (empty string) causes an uncaught `TypeError` that escapes the tool error handler

**File:** `packages/emmy-tools/src/web-search.ts:176` and `packages/emmy-tools/src/web-search.ts:184`

**Issue:** The URL construction on line 176 is executed before the `try { ... } catch { ... }` block that starts at line 184. The `??` (nullish coalescing) operator on line 106 (`process.env.EMMY_SEARXNG_URL ?? "http://127.0.0.1:8888"`) only short-circuits on `null` or `undefined`. An empty string `""` passes through, giving `cfg.baseUrl = ""`. `new URL("/search", "")` throws `TypeError: Invalid URL` synchronously before entering the try block, so the error is not caught by the fallback handler and propagates as an unhandled rejection to the agent loop.

The default when `EMMY_SEARXNG_URL` is unset is safe (`http://127.0.0.1:8888`). This only triggers if an operator explicitly sets `EMMY_SEARXNG_URL=""`.

**Fix:**
```typescript
// At line 106 in _resolveDefaultBaseUrl():
function _resolveDefaultBaseUrl(): string {
    const v = process.env.EMMY_SEARXNG_URL;
    return v && v.trim() ? v.trim() : "http://127.0.0.1:8888";
}
```
Or alternatively, wrap the URL construction in the existing try block:
```typescript
let url: URL;
try {
    url = new URL("/search", cfg.baseUrl);
} catch {
    return _buildError("web_search failed: invalid SearxNG base URL configured.");
}
```

### IN-01: `start_emmy.sh` uses `eval` to expand `DOCKER_RUN_ARGS`; pre-existing pattern flagged for traceability

**File:** `scripts/start_emmy.sh:127`

**Issue:** Line 127 uses `eval docker run --name emmy-serve --detach "$DOCKER_RUN_ARGS"` (with `# shellcheck disable=SC2086`). `DOCKER_RUN_ARGS` is the output of a Python subprocess (`render-docker-args`). This is a pre-existing pattern from Phase 1/4; Phase 04.2 adds the `--install-sidecar-unit` branch but does not touch the `eval` line. If a malicious or corrupted profile's `serving.yaml` caused `render-docker-args` to emit shell metacharacters in its output, `eval` would execute them.

No fix required in Phase 04.2 scope (this is Phase 4 code). Tracking here because the pattern is worth addressing in a future hardening pass: replace `eval` with `mapfile -t args_array < <(...)` and `docker run "${args_array[@]}"`.

### IN-03: `renderSidecarStatus` `error=` segment truncation test has an off-by-one in the assertion

**File:** `packages/emmy-ux/test/slash-commands.start-stop-status.test.ts:185-186`

**Issue:** The test at line 185 extracts the text after `"error="` and asserts `errSegment.length <= 40`. However, `renderSidecarStatus` does `s.last_error.msg.slice(0, 40)` (40 chars of the message content), then prepends `"error="` (6 chars). The resulting full segment is `error=<up-to-40 chars>` which appears within the space-separated parts string. The test measures `text.slice(errIdx + "error=".length)` but this includes everything after the `error=` segment in the full status string — any trailing parts (none in this test case since `last_error` is the last field) would inflate `errSegment`. In the test the `last_error` field is last so there are no trailing parts, making the assertion accidentally pass. The assertion should compare `errSegment.split(" ")[0]?.length <= 40` to be precise.

No runtime impact. The `slice(0, 40)` in production code is correct. This is a test assertion fragility issue.
